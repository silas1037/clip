 一些关于吉里吉里的XP3文件加解密研究（主要用于FHA汉化）

一些缩写：
krkr：吉里吉里
xor：异或
^：异或运算符
FSN：Fate／stay night
FHA：Fate／hollow ataraxia
fate：包括上面2个游戏（专指PC版）
TM：TYPE-MOON
FC：Fate文本文件破解程序v0.5测试版，http://bbs.sumisora.net/read.php?tid=222058

一、krkr的初始化流程：
1.游戏的执行程序启动，查找.tpm文件。若有，则用这个文件解密，否则进入下一步。
2.载入startup.tjs文件，这个文件默认只有一个作用，就是载入Initialize.tjs文件。Initialize.tjs则进行初始化，设置搜索路径并搜索该文件夹下所有的.xp3文件，若有patch文件，则依次加载patch.xp3、patch2.xp3、patch3.xp3……（个数未限制，不超过整型范围就行了）。
3.经过几番加载后，最终调用first.ks运行游戏。

其中最关键的就是第1步。与未加密的游戏不同，fate通过cxdec.tpm文件解密所有的.xp3文件，然后再运行。
另外第2步也给出了汉化补丁的做法，即将汉化后的剧本等资源放在patch.xp3里。若已有patch，且需要原有的patch，则放在最后1个patch名+1的.xp3里，如：已有patch.xp3、patch2.xp3、patch3.xp3，则放入patch4.xp3里。


二、krkr的加密方法：
1.编写一个xp3enc.dll（其实如果FHA放出这个文件的话，一切就好办了），里面的XP3ArchiveAttractFilter_v2函数描述了需要加密的.xp3文件的加密算法。

该算法可以对文件的任意位置进行加密，但加密某1字节时，不能依赖于其他字节。基本就是只能对某字节进行一系列运算（并且能找到逆运算，否则无法解密。关于逆运算，理解为+1和-1之间的关系吧）。最常用的加密就是xor了，fate的加密就只用了xor。

关于xor，再补充说明下：
对于1位2进制数值进行xor，就是比较2者是否相同，相同为0，不同为1。如0 ^ 0和1 ^ 1的结果都是0，1^ 0和0^1的结果都是1。
文件可以看成很多个字节组成的。在x86结构下，一个字节是8位，即每个字节可以是0~255（共2的8次方个）之间的数。
对于2个字节，如123、234，转换为2进制为01111011b和11101010b（后缀b表示2进制）。则它们之间异是将每一位分别异或。因此 123 ^ 234 = 01111011b ^ 11101010b = 10010001b = 145。
异或的逆运算也很简单，123 ^ 145 = 234，234 ^ 145 = 123，有兴趣可以自己算算。（PS：windows自带的计算器就可以进行进制转换和xor运算。）

关于xp3enc.dll的详细编写方法与破解无多大关系，有兴趣的可参考Miliardo的《关于加密插件的编写》一文：
http://kcddp.keyfc.net/bbs/viewthread.php?tid=272

2.将xp3enc.dll放在krkrrel.exe所在的文件夹（需要krkr或KAGeXpress的开发包，在tools文件夹下）。
运行krkrrel.exe，选择要打包成.xp3的文件夹，并勾选“使用xp3enc.dll”。
最后选择输出文件名，按下“生成”按钮就生成一个加密的.xp3文件了。


三、krkr的解密方法：
1.编写一个.tpm文件，一般可命名为xp3dec.tpm，FHA是用cxdec.tpm。一般不用去管这个名字。
这个文件其实也是个.dll文件，里面的TVPXP3ArchiveExtractionFilter函数描述了与xp3enc.dll对应的解密算法。如加密算法是与54异或（即FSN的加密算法），则解密算法也是与54异或。

关于.tpm文件的详细编写方法可参考上述《关于加密插件的编写》一文。

2.运行游戏时，krkr自动将该.tpm文件用于解密。参见“一、krkr的初始化流程”。


四、FSN的破解和汉化方法：
1.将.xp3文件里的剧本等资源提取出来（剧本都在data.xp3下）。

这个有很多办法，最简单的就是用ExtractData，截止目前为止最新版本为1.04 [2007/06/26]。
主页：
http://atelibet.s11.xrea.com/side_yuu/tool/
下载连接：
http://atelibet.s11.xrea.com/side_yuu/tool/ExtractData/ExtractData104.7z

从1.02版以后，ExtractData已经能自动根据.tpm文件解密.xp3文件了。需要将.tpm文件和.xp3文件放在同一文件夹下，若无效则勾选“设定－抽出设定－简易复号机能有效”。（看不懂日文的按ctrl+e，选上面第2个选项页，再勾选第3个即可。）经试验只对简单的加密有效，不过FSN已经能自动破解了＝。＝

1.01版之前解压出来的剧本是加密过的，打开是乱码，可以用我写的FC破解。
原理很简单，所有字节和54异或即可。

另外也可以用susie＋ax_xp3.spi解压，比较麻烦，解压后也是加密的，不叙述了。

注意，解密后的剧本文件是日文编码，可以用IE打开，转换到日文编码查看。

2.将剧本中的日文剧情翻译成中文：
注释和代码就不用翻译了，这个汉化组应该能区分吧＝。＝

翻译完后要保存文件，文件名是相同的。为了让其他编码的系统能正常阅读，可以保存为unicode格式（用记事本的话，选择“文件－另存为”，将编码改为unicode，文件名不变）。一般来说改为unicode格式后文件会变大，因为很多原来只用一个字节表示的字符（ASCII字符，如英文字母）将使用2个字节表示。

3.加密剧本：
为了能让FSN正常解密，必须将剧本加密，才能让其解密后还原为正常剧本。

加密方式也很简单，用上述的FC再次解密（因为异或和其本身互为逆运算，所以再次解密实际相当于加密），使用默认的54异或即可。

4.打包成.xp3文件：
对于已经加密的文件，可以直接用krkrrel.exe打包。
若有fate的xp3enc.dll文件的话（不排除你入侵TM，不过直接盗取未加密的资源更好……），可以跳过上面的第3步，直接在打包时加密（参见“二、krkr的加密方法”）。
为了减小补丁的大小，只将汉化的文件打包为patch.xp3（或patch2.xp3……）即可（需对应相应文件夹）。

5.把patch.xp3放在游戏目录下，运行游戏，检验汉化效果。


五、FHA的破解和汉化可能性：
终于到这里了，很遗憾地告诉大家，FHA的破解异常复杂，我没有试验过，这里只分析下。

1.FHA的加密方式：
同样还是用异或，但和FSN不同的是，FHA的文件是分块加密的，即每块的异或值是不同的。具体地来说就是，每个文件的异或值是不同的，甚至一个文件的几个部分异或值都不同（遗憾的是FHA的大部分文件都是如此）。

这个就是一切麻烦的根源，原因稍后解释。

另外，每个.xp3文件都附带了一个.sig文件，里面记录了SHA256/PSS/RSA的公钥。「Fate／hollow ataraxia」インストールチェッカ.ini文件里也记录了PUBLIC KEY。
目前尚未知.sig文件是否与游戏运行无关，也许只是「Fate／hollow ataraxia」インストールチェッカ.exe用来校验.xp3文件是否损坏或修改了。
若不是的话，还得为自己写的patch.xp3附加patch.xp3.sig。但我估计这个可能性很小，因为TM自己也可能发布补丁，不会弄得太麻烦。

2.FHA的解密方式：
同样还是用游戏执行文件调用cxdec.tpm来解密。
若能知道里面的算法就很容易破解了。

3.FHA的剧本文件破解：
音乐、图片、动画等与汉化无直接关系的多媒体资源的破解就不说了，只说剧本文件。
先把data.xp3解压了，方法同FSN的破解。

用FC可以找到一般的剧本文件的异或值，并进行破解。（破解的算法见文末附录。）
对于大于800字节的文件，异或值可能有多个（剧本文件一般不超过2个）。

破解时请记录使用的异或值。若有多个异或值，还需将解密的几个文件沿乱码处拼接在一起，并记录偏移量（即发生乱码处与第1个字节之间的字节个数，用16进制编辑器可以直接查看）。

然后就可以汉化了。

另一种可能性：
因为krkr是开源的（不过我没找到源代码），研究下源代码应该能找到调用cxdec.tpm的方式。将其添加一句输出解密后的文件应该也不难。
FHA貌似还有unit test，可以自动跑一遍剧本，正好方便输出……

4.FHA的剧本文件的加密：
还是那句老话，若有FHA的xp3enc.dll文件的话，同上……

若剧本文件只有1个异或值，直接再次异或即可。若有2个，将文件分别异或2次，并沿偏移量处将2个文件加密后的内容拼接到一起。
当然，也可以写个程序，输入异或值和偏移量自动完成这个任务。有兴趣的可以自己试试，修改我的程序添加这个功能也很简单（对于会C++的人来说）。

5.FHA的剧本打包：
应该来说，直接打包成data.xp3就行了。

不过据说有人解压了data.xp3，然后未经修改直接打包，替换后游戏报错。
这可能有2个原因：
1.和FHA使用的打包程序和选项不同。
FHA的.xp3中有个“$$$ This is a protected archive. $$$ 著作者はこのアーカイブが正規の利用方法以外の方法で展開されるこ”文件，若是用中文版的打包程序的话，这个文件是中文的……
尝试用日文版，组合一下各个选项打包成和原来相同的.xp3文件。
2.FHA修改了.xp3文件格式和krkr执行程序。krkr会检查.xp3格式是不是其修改过的格式。
解决办法嘛，同上打包后，用16进制的编辑器比较2个.xp3文件的差异，按原版的.xp3文件进行修改。

另外，因为汉化后更改了文件大小，可能也会影响正常解密。为了节约空间可能需要打包成patch.xp3。但必须提供patch.xp3的解密方式，这2个原因都需要了解加解密算法。

若能正常运行的话，打包成data.xp3也无所谓了。否则必须了解加解密算法。下面给出几种方法：
1.多破解一些剧本，分析文件大小、偏移量、异或值是否有什么联系，相邻的剧本间异或值是否有联系，然后推出算法，不过这仅限于算法比较容易推出的情况。
2.反汇编或跟踪cxdec.tpm，找到TVPXP3ArchiveExtractionFilter函数，分析算法。
我虽然还算懂汇编语言，但没跟踪过dll文件，静态反汇编也不知道该怎么找＝。＝
有强人的话可以试试。
3.分析krkr的源代码，找到调用cxdec.tpm的方式，做个程序直接调用cxdec.tpm来解密（根据异或的对称性，再次解密的话就是加密了）。
这个算是比读汇编代码容易多了，但偶没研究过……

感觉这些方法都好麻烦，但愿可以直接打包成data.xp3。


附录：

一、FC的破解算法：

1.头部的破解：
如果学习了怎样制作krkr的游戏的话，应该知道剧本的开头字符只能是/、;、*、@和[。
其中/后应紧跟*或/组成注释。若是的话则很可能头部为/，并将其可能性设为最高。
;可以单独作为注释。
*是标号，后面的字符中可能会有|字符表示章节名。若有的话增加可能性（越近可能性越高）。
@是标记指令用的。
[也用于标记指令，后需跟]表示指令结束。若有的话增加可能性（越近可能性越高）。
通过假定开头字符是这几个，分别分析其可能性，然后按可能性排序即可。

2.尾部的破解：
为了可读性，文本文件一般都会换行的。换行在windows平台下是用"\r\n"，16进制为0D 0A，Unicode则是00 0D 00 0A（限于汉化版，不讨论）。
这样，只要连续2个字节的异或值等于0x0D ^ 0x0A，则可能是换行（只是可能）。
然后从文件尾开始向前输出可能的异或值即可，一般最后3个中会有正确的异或值。


二、FC的源代码（C++）：

//有很多乱七八糟的注释，都是以前的研究成果，懒得删了，将就看吧
//其他除了帮助代码删了，其他未改动

#include <iostream>
#include <fstream>
#include <sstream>
#include <algorithm>

using namespace std;

namespace
{
    //从主程序参数arg中得到数据，数据保存在result
    template<typename T>
    void getArg(T& result, const char* const& arg)    //不检查参数是否给错，这个由调用程序控制
    {
        stringstream buf;
        buf << arg;
        buf >> result;                //貌似无法用buf的good、bad判断输入出错的情况，所以不检查
    }
////////////////////////////////////////////////////////////////////////////////////////////////
    //在buf指向的区域内查找回车的可能值，并将值保存在temp指向的缓冲区中，并返回找到的数目
    template<typename T1, typename T2>
    int findCR(T1* &buf, T2* &temp, unsigned int bufSize)
    {
        int count = 0;
        for (int index = 0; index < bufSize; ++index)
        {
            if ((buf[index] ^ buf[index + 1]) == (0x0D ^ 0x0A))    //回车与换行的ASCII码，貌似无法用'\r'和'\n'
            {
                temp[count++] = buf[index] ^ 0x0D;
            }
        }
        return count;
    }
}
////////////////////////////////////////////////////////////////////////////////////////////////
int main(int argc, char *argv[])
{
    try
    {
////////////////////////////////////////////////////////////////////////////////////////////////
        //判断是否输入文件名
        if (argc < 2)   
        {
            cerr << "请输入需要破解的文件名！\n"
            "格式：fatecrack [路径]需要破解的文件名 [异或值] [检测字符长度]\n"
            "详细帮助请使用\"fatecrack /?\"查看。\n";
            return 1;
        }
////////////////////////////////////////////////////////////////////////////////////////////////
        //显示帮助
        if(string(argv[1]) == "/?")
        {
            //输出帮助信息，这里就不放出来了……
            return 0;
        }
////////////////////////////////////////////////////////////////////////////////////////////////
        //读取源文件
        ifstream in(argv[1], ios::binary);
        //必须是2进制，否则不能完全破解或出现乱码，下同
        if (!in)
        {
            cerr << "源文件打开失败！\n";
            return 2;
        }
////////////////////////////////////////////////////////////////////////////////////////////////
        //获取异或值
        unsigned char FSN_XOR = 0x36;            //Fate / stay night的默认异或值0x36
        unsigned int xorValue = FSN_XOR;        //存放异或值，默认为FSN的
        if (argc > 2)
        {
            getArg(xorValue, argv[2]);

            if (xorValue > 0xFF)            //判断异或值输入是否超过访问；貌似无法用buf的good、bad判断输入出错的情况，所以忽略
            {
                cerr << "异或值输入错误！\n";
            }
        }

////////////////////////////////////////////////////////////////////////////////////////////////
        /*
        查找异或值，使用分别查找头尾的异或值的方式

        这段算法不够好，不过更详细的方法太繁琐，且得到结果太多，还不如手动破解。
        脚本文件头（.ks和.tjs）一般是注释或命令开头，即可能是/、;、*、@和[，但其他文件则没限定，所以可以区分对待。
        文件结尾一般会有换行，所以检测到的最后一个可能是回车的字符一般就是回车了。
        更省事的办法是直接检验最后2个字符，貌似要定位到文末。
        */
        if (xorValue == 0)
        {
////////////////////////////////////////////////////////////////////////////////////////////////
            //获取读取字符数
            const unsigned int MAX_SIZE = 100;    //默认读取的字符数
            int maxSize = MAX_SIZE;
            if (argc > 3)
            {
                getArg(maxSize, argv[3]);
                if (maxSize <= 0)
                {
                    cerr << "检测字符长度数输入错误！\n";
                    maxSize = MAX_SIZE;
                }
            }
////////////////////////////////////////////////////////////////////////////////////////////////
            //头部处理

            //读取头部
            char* headBuf = new char[maxSize];
            in.read(headBuf, maxSize);
////////////////////////////////////////////////////////////////////////////////////////////////
            //查找脚本文件可能的异或值

            //定义头字符
            unsigned int HEAD_CHAR_NUM = 5;        //脚本的头字符只有5个：/、;、*、@和[
            pair<unsigned char, int> xorHead[HEAD_CHAR_NUM];
            //2个值分别是异或值和权值，后者用来排序异或值
            xorHead[0].first = headBuf[0] ^ '/',
            xorHead[1].first = headBuf[0] ^ '*',
            xorHead[2].first = headBuf[0] ^ ';',
            xorHead[3].first = headBuf[0] ^ '@',
            xorHead[4].first = headBuf[0] ^ '[';
////////////////////////////////////////////////////////////////////////////////////////////////
            //用'/'构成的注释后肯定接'/'或'*'，若是则可能性最高，否则舍弃
            if ((headBuf[1] ^ xorHead[0].first) == '/' || (headBuf[1] ^ xorHead[0].first) == '*')
            {
                xorHead[0].second = 200;    //设为最大可能，读取字数有限，不检查“*/”
            }
            else
            {
                xorHead[0].second = -1;
            }
////////////////////////////////////////////////////////////////////////////////////////////////
            //除'/'外其他字符的权值设定，以下一个回车离头字符的距离为权值
            char* last = headBuf + maxSize;
            for (int index = 1; index < HEAD_CHAR_NUM; ++index)
            {
                xorHead[index].second = last - find(headBuf, last, xorHead[index].first ^ 0x0D);    //0x0D为回车
            }
