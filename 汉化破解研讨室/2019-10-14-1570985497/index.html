 SiglusEngine引擎破解手记——附脚本工具SiglusTools

做一个破解记录，免得下次破解的时候忘了

该记录以《銀色、遥か》的破解为例。
只有新版无壳的SiglusEngine可以用这种方法破解，老版带壳还屏蔽Debug，我无法处理……

1、系统语言检测：
这部分破解是最后做的，因为以前都是用AlphaROMdiE。但最新版引擎AlphaROMdiE的Dummy检测破解出了问题。既然Dummy得手动破解，干脆不用AlphaROMdiE。

虽然是用OD破解，但其实我并不懂多一点。破解这个走了不少弯路。GetUserDefaultLCID、GetTimeZoneInformation啥的下了不少断点。然而一个都没断到。
最后选择了比较粗暴的方法。
OD挂载程序之后，用Ctrl+F8自动步过。自动步过遇到CALL会直接运行过去，所以当弹出“Japan Only”的时候，在停住的那个CALL下断点，重新开始。运行到这个断点的时候F7跳转进这个CALL，再Ctrl+F8继续自动步过。
重复这一过程，观察每次的CALL，如果遇到断在MessageBox，说明跳过头了。
Copy code
015E0658  |. 6A 30          PUSH 30                                  ; /Style
015E065A  |. 0F434D D8      CMOVNB ECX,DWORD PTR SS:[EBP-28]         ; |
015E065E  |. 8D45 C0        LEA EAX,DWORD PTR SS:[EBP-40]            ; |
015E0661  |. 837D D4 08     CMP DWORD PTR SS:[EBP-2C],8              ; |
015E0665  |. 51             PUSH ECX                                 ; |Title
015E0666  |. 0F4345 C0      CMOVNB EAX,DWORD PTR SS:[EBP-40]         ; |
015E066A  |. 50             PUSH EAX                                 ; |Text
015E066B  |. FF35 CC5CDC05  PUSH DWORD PTR DS:[5DC5CCC]              ; |hOwner = NULL
015E0671  |. FF15 C0D49C01  CALL DWORD PTR DS:[<&USER32.MessageBoxW>]; \MessageBoxW

那么，回到上一个CALL的断点，很可能就是系统检测的入口。
Copy code
015DEE94   . E8 37170000    CALL SiglusEn.015E05D0
015DEE99   . 84C0           TEST AL,AL
015DEE9B   . 0F84 25070000  JE SiglusEn.015DF5C6

尝试NOP掉这个CALL：
Copy code
015DEE94     90             NOP
015DEE95     90             NOP
015DEE96     90             NOP
015DEE97     90             NOP
015DEE98     90             NOP
015DEE99   . 84C0           TEST AL,AL
015DEE9B   . 0F84 25070000  JE SiglusEn.015DF5C6

成功启动游戏，破解成功！
Quote:

第一处的
Copy code
015DEE94   . E8 37170000    CALL SiglusEn.015E05D0
015DEE99   . 84C0           TEST AL,AL
015DEE9B   . 0F84 25070000  JE SiglusEn.015DF5C6

要注意TEST AL,AL和JE是在检查CALL的返回值。如果仅NOP掉CALL的部分，那么此处AL的值是不确定的，如果不清除AL的数值或把JE变成JMP，那么这里NOP掉CALL能过仅仅是运气好。

根据大佬的提示，这里为了保险起见需要注意一下AL的值。
如果这里將AL的值改为0，进程会直接结束，说明下面这个JE是跳转到结束程序的地方。所以：
Copy code
00C2EE94     B0 01          MOV AL,1
00C2EE96     90             NOP
00C2EE97     90             NOP
00C2EE98     90             NOP
00C2EE99     84C0           TEST AL,AL
00C2EE9B     0F84 25070000  JE SiglusEn.00C2F5C6

这样就可以了。
另外也可以干脆把TEST和JE这两行也NOP掉……

2、Dummy检测破解
这一步是最先搞的，也是费了不少周折。开始用了一种方法找到了，但其实用上面破解系统检测同样的方法找来的更简单。不过还是记录一下。


最初是在CreateFile下断点，发现Dummy检测的部分并没有断。于是去Windows API手册里找其他和检查文件有关的命令。几番尝试之后，确定使用的是GetFileAttributes。然后前前后后查看了半天，找到了检测驱动器GetDriveType。
Copy code
00ED6988   . 8B3D F4D12801  MOV EDI,DWORD PTR DS:[<&KERNEL32.GetDriveTypeW>]  ;  KERNEL32.GetDriveTypeW

注意这里并不是CALL，顺着找下去，找到CALL EDI：
Copy code
00ED6A88   . FFD7           CALL EDI                                          ;  KERNEL32.GetDriveTypeW
00ED6A8A   . 83F8 05        CMP EAX,5
00ED6A8D   . 75 43          JNZ SHORT SiglusEn.00ED6AD2

CMP EAX,5游戏会选择返回值是5（CD-ROM）的驱动器检查文件。
本来打算从这里下手，但是左思右想也只想到了把Dummy文件放在本地磁盘里进行检测的方法。没有太大意义，还是需要整个跳过才有意义。所以还是要找入口。

加载光盘镜像。或者：
临时将上面那里改成CMP EAX,3变成选择返回值是3（本地磁盘）的驱动器。然后把光盘里Dummy检测相关的文件（env文件和dummy文件夹下dummy.001文件）放在本地磁盘下。
然后在MessageBox下断点，停在“感谢购买”的窗口处：
Copy code
00E8333B  |> 57             PUSH EDI                                    ; /Style
00E8333C  |. 50             PUSH EAX                                    ; |Title
00E8333D  |. 56             PUSH ESI                                    ; |Text
00E8333E  |. FF35 CC5C6805  PUSH DWORD PTR DS:[5685CCC]                 ; |hOwner
00E83344  |. FF15 C0D42801  CALL DWORD PTR DS:[<&USER32.MessageBoxW>]   ; \MessageBoxW
Copy code

0590E638   0FF04428  |Text = "この度は『銀色、遥か』をお買い上げいただき、ありがとうございます。"

0590E640   00000040  \Style = MB_OK|MB_ICONASTERISK|MB_APPLMODAL

F8步过到RETN后，观察返回位置上一行的CALL。在这一行CALL下断点，重新运行，发现这个CALL已经在开始Dummy检测之后。
继续F8步过到下一个RTEN后。
Copy code
00DDE36D  |. 50             PUSH EAX
00DDE36E  |. E8 8D640B00    CALL SiglusEn.00E94800
00DDE373  |. E9 14080000    JMP SiglusEn.00DDEB8C

返回到了JMP这一行，在上面那个CALL下断点，重新运行。可以看出这个CALL在弹出窗口之前。
尝试NOP掉这个CALL：
Copy code
00D9E36D     50             PUSH EAX
00D9E36E     90             NOP
00D9E36F     90             NOP
00D9E370     90             NOP
00D9E371     90             NOP
00D9E372     90             NOP
00D9E373  |. E9 14080000    JMP SiglusEn.00D9EB8C

成功跳过Dummy检测，破解成功！
*这里可能会抛出异常，估计是Logo动画的原因，Shift+F9应该是可以忽略的。


另一种方法就是和前面一样，Ctrl+F8自动步过，遇到弹窗了的CALL就断点，重新运行。F7步进，再继续Ctrl+F8自动步过。
最后会停在上面那个MessageBox的地方。那么前一个断点就是Dummy检测的入口。
*中途可能会遇到卡在某个CALL的情况。这时F12暂停，再F9运行就会运行到那个CALL下面一行。再接着Ctrl+F8就行了。
*也有会循环很多次的代码，可以在循环之后临时下个断点，F9运行到断点再Ctrl+F8继续。


3、字体选择：
引擎默认只能选择日文字体，即使用各种方法强制使用中文字体，也会变成日文默认字体。
有一个简单的办法就是自己准备一个同时支持中文和日文的字体。新版引擎有一个装载自带字体的功能，两个字体存在dat文件夹下，装载是在Gameexe.dat内定义的。可以用自体编辑器自己弄一个同时用中文和日文代码页的字体放在dat文件夹下，然后修改Gameexe让游戏装载，可以省去单独安装字体的过程。
当然，如果想让玩家自己选择中文字体，就需要修改游戏了。
最初我在CreateFont和CreateFontIndirect下断点前后找了好久，一直没什么收获。不过发现了一段存储着看起来是字体信息的内存：

从中随便挑一个，能看到在字体名字前面有一个数是86，这是中文字符集134。显然，游戏会检查这个数，然后过滤掉不是80（日语字符集）的字体。所以在这个86上下内存断点。F9运行。停在这里：
Copy code
00E751C1  |> 85C0           /TEST EAX,EAX
00E751C3  |. 74 0B          |JE SHORT SiglusEn.00E751D0

00E751D6  |. 05 1C010000    |ADD EAX,11C

REP MOVS这里看上去是在把内存转存，还没有进行判断，那么我们删除内存断点，跟踪EDI重新下内存断点。继续执行，反复几次之后断在了这里：
Copy code
00E73A70  |> 807E 17 80     /CMP BYTE PTR DS:[ESI+17],80
00E73A74  |. 75 08          |JNZ SHORT SiglusEn.00E73A7E
00E73A76  |. 56             |PUSH ESI                                              ; /Arg1
00E73A77  |. 8BCB           |MOV ECX,EBX                                           ; |
00E73A79  |. E8 220B0000    |CALL SiglusEn.00E745A0                                ; \SiglusEn.016645A0
00E73A7E  |> 81C6 1C010000  |ADD ESI,11C
00E73A84  |. 3B73 34        |CMP ESI,DWORD PTR DS:[EBX+34]
00E73A87  |.^75 E7          \JNZ SHORT SiglusEn.00E73A70

显然，这里进行了判断，如果不是80就跳过。那么，尝试将这里改成86。
重新启动游戏，成功。
（其实从ESI+17就能看出这个位置没错，因为从字体的结构体LOGFONT来看，字符集这一位前面有5个Long和3个Byte，正好是0x17）
PS：如果将这行JNZ给NOP掉，就可以让游戏载入全部的字体。但有个问题就是支持多个字符集会在字体选择那里显示多个（虽然选哪个都一样）。看着不舒服……不过因为有很多中文日文字体会支持默认字符集，所以如果把80改成0就可以选择这些字体，间接达到同时可选中日文字体的效果（当然，除了系统默认的字体往往都不支持这个方法）。

不过同时，GBK没有的日文符号（比如日文圆点、音符）的间距反而会出问题，所以汉化的时候需要注意。

接下来是文件处理，这个道理也很简单，主程序里对应的文件名（扩展名）改掉就可以了。
主要需要改的是Scene.pck和Gameexe.dat，因为长度只能短不能长，所以一般来说改成Scene.chs和Gameexe.chs。
另外，这两个文件名的字符串在文件里多处，比如上面那段代码里就有，前后都是文字，显然是提示框内的内容，所以这里不是要改的地方。具体要改的是哪个自行判断一下吧。
如果要给视频加字幕，需要改视频文件扩展名。因为SiglusEngine调用Window Media Player播放视频，所以这里可以取巧。
游戏一般是WMV或者MPG格式，所以加了字幕的视频压制成MP4就可以和源文件不冲突。

可以猜到，这WMV、MPG、AVI几个格式是按优先顺序读取的，因为都是调用WMP所以不需要担心不同格式的差别。
把WMV改成MP4，AVI改成WMV（如果游戏确实有AVI的格式的视频，就改游戏没有的那个）就行了
像g00修图，如果改掉的话（比如改成g01），游戏就只会读g01，那样就必须所有图片都弄个g01的，显然不现实。
但因为图片格式也有好几种，所以也有取巧的办法，不过和视频不同，不同格式读取不一样，简单改了会出问题。所以后面详细讲怎么改。
不过，像dbs文件这种，只有部分需要翻译，但文件数量很少的尺寸也很小，可以改掉扩展名，然后连不需要翻译的文件也一起放进汉化补丁里。

就不全列出来了，只放一个参考和一个要改的。

dds这个格式似乎从来没用过。那么，我们可以考虑把g00改成g01，然后把dds这个改成g00。
但这里有两个问题，一个是“g00”那个字符串，还用在了定位g00的目录上，如果直接改了，会因为找不到g01目录而出问题。
另一个是dds改成g00以后，因为格式对应不上所以游戏也会报错。
我们先来解决第二个问题。观察到除了“g00”，其他几个都有一行MOV EBX,?，而且是2、3、4、5顺着下来的。
这么看来，g00对应的很可能是MOV EBX,1。在“g00”字符串后面的LEA EBX,DWORD PTR DS:[EAX+1]下断点，让程序运行到这里。F7步入下一行之后，EBX的值确实变成了1。说明这个猜测是正确的。
那么我们把“dds”改成“g00”，把下面的5改成1就可以了：
Copy code
00CB425E  |> 68 CC901001    PUSH SiglusEn.011090CC                   ;  UNICODE "g00"
00CB4263  |. 8D4D C0        LEA ECX,DWORD PTR SS:[EBP-40]
00CB4266     BB 01000000    MOV EBX,1
00CB426B  |. E8 106FF2FF    CALL SiglusEn.00BDB180

接下来处理改“g01”的问题。既然因为用了同一个字符串冲突了，那我们就在别的地方再准备一个字符串。
首先我们在内存映射中找到主程序的.rdata部分。然后从中找到一个有一大片空白的地方：
Copy code
010C3C50  5F 79 6E 00 5F 6C 6F 67 62 00 00 00 5F 6E 65 78  _yn._logb..._nex
010C3C60  74 61 66 74 65 72 00 00 00 00 00 00 00 00 00 00  tafter..........
010C3C70  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00  ................
010C3C80  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00  ................

为了保证不会冲突，我们需要空出几个00，然后填上“g01”：
Copy code
010C3C50  5F 79 6E 00 5F 6C 6F 67 62 00 00 00 5F 6E 65 78  _yn._logb..._nex
010C3C60  74 61 66 74 65 72 00 00 00 00 00 00 00 00 00 00  tafter..........
010C3C70  67 00 30 00 31 00 00 00 00 00 00 00 00 00 00 00  g.0.1...........
010C3C80  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00  ................

（注意ASCII和Unicode的区别）
然后把那个PUSH的地址改成这里：
Copy code
00CB3FED     68 703C0C01    PUSH SiglusEn.010C3C70                   ;  UNICODE "g01"

（注意，这里保存的时候会提示“请确定更新重定位”，点“是”就行了）
接着在主程序里找到刚才修改的内存的对应位置，同样填上“g01”：
Copy code
005F2250  5F 79 6E 00 5F 6C 6F 67 62 00 00 00 5F 6E 65 78  _yn._logb..._nex
005F2260  74 61 66 74 65 72 00 00 00 00 00 00 00 00 00 00  tafter..........

005F2280  00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00  ................
