恩，是差不多的
不过偶这方法在有明确要断的API函数时保险性更高
因为很多情况下程序调用的API不会直接在输入参考中给明的，比如call edx这种命令很可能执行到这一句时，edx的值才指向API函数
而用偶这方法不管是EXE本身还是他载入的DLL，只要是本进程中的线程调用了你要断的API函数就绝对能断下来

缺点是要找到调用的call地址要从右下角的窗口里跳过去，而且要直观的看到参数必须断2次
跳过去之后再在那个call上下断点，那么下次再执行到这个call时就会看到函数名和传递的参数了

关于字体名的问题，并不是调用的字体必须日文字体，只要包含日文编码即可
而如果你是要汉化，要改的也并不是字体名，而是char字段

你给出的数据并不是标准的LFont结构，建议你跳到
77F1B752    E8 8C2BFEFF    CALL GDI32.CreateFontIndirectA
这个地址再下断，然后重新运行一次
在这里中断后，右下窗口的第一项就是Lfont结构的一个地址指针了，查看这里的内存就能看到一个LFont结构，如果是日文游戏的话，里面应该会有个16进制值为80的地方，改成86就是简体中文了
但是改这里是没用的，下次运行又会还原了，你必须找到构建这个Lfont结构的地方并修改相应的赋值命令才行

修改程序总是很多偷懒的办法...针对LOGFONT的初始化也有.

这LOGFONT是一个struct,在ANSI下实际上是LOGFONTA.
LOGFONT声明如下:

typedef struct tagLOGFONT {
    LONG lfHeight;        // 4 bytes,
    LONG lfWidth          // 4 bytes,
    LONG lfEscapement;    // 4 bytes,
    LONG lfOrientation;  // 4 bytes,
    LONG lfWeight;        // 4 bytes,
    BYTE lfItalic;        // 1 byte,
    BYTE lfUnderline;    // 1 byte,
    BYTE lfStrikeOut;    // 1 byte,
    BYTE lfCharSet;      // 1 byte,
    BYTE lfOutPrecision;  // 1 byte,
    BYTE lfClipPrecision; // 1 byte,
    BYTE lfQuality;      // 1 byte,
    BYTE lfPitchAndFamily;// 1 byte,
    TCHAR lfFaceName[LF_FACESIZE];
} LOGFONT, *PLOGFONT;

在每个成员的后面标注上了长度.各成员在struct中的偏移量并不总是...在不同情况下编译出来的偏移量其实不是一定的.下面会举个例子.

实际使用的时候,很少会一开始就一口气把LOGFONT的成员全部赋好值.一般只是设定一些自己的程序必需的,例如这样:

Copy code
LOGFONT logFont;

logFont.lfStrikeOut = 0;
logFont.lfUnderline = 0;
logFont.lfHeight = 42;
logFont.lfCharSet = 0x086; // 指定使用GBK字符集(准确说是CodePage 936)
logFont.lfEscapement = 0;
logFont.lfItalic = TRUE;
//...
font = CreateFontIndirect(&logFont);

中间说不定还会有EnumFontFamiliesEx之类的函数,所以连字体的FaceName都未必会一开始就在LOGFONT里指定.

既然是个struct,各成员在内存里肯定是连续分配的.
因此,在给LOGFONT的成员赋值的时候,也比较容易编译出这样形式的代码:
mov ptr [R32+offset], val
也就是说使用base+offset形式的间接寻址方式来赋值.

让我们来看看一段简单的C代码:
Copy code

00401070  |.  FF15 00B04000     call dword ptr ds:[<&GDI32.CreateFontIndirectA>]   ; \CreateFontIndirectA


我想说什么呢? 编译器优化会使这LOGFONT的参数辨别起来有点小困难.但是形式总是一样的.
针对CharSet这个参数,我们要寻找的就是mov byte ptr [R32+offset], val形式的命令.
假如说游戏原本是以日文显示的(表现就是不以AppLocale或别的引导程序启动游戏,文字也能以日文正确显示),那么我们要寻找0x080这个参数.在LOGFONT的赋值上,表现为:
mov byte ptr [R32+offset], 80

举例的话,可以看看Nitro+的游戏里,system.dll里的一小段:
Copy code
C740 04 E0FFFFFF   mov dword ptr ds:[eax+4],-20
C740 14 90010000   mov dword ptr ds:[eax+14],190
C640 1B 80         mov byte ptr ds:[eax+1B],80      ; 注意这里
C640 1C 02         mov byte ptr ds:[eax+1C],2
C640 1D 01         mov byte ptr ds:[eax+1D],1
C640 1E 03         mov byte ptr ds:[eax+1E],3
C640 1F 34         mov byte ptr ds:[eax+1F],34

这里,eax是指向一个LOGFONT实例的指针.

怎么找呢? DLL文件并不一定要运行原程序来加载,可以通过OllyDbg直接调用LoadDLL.exe来加载.这里Phantom里的system.dll并没有加壳,可以直接处理.
汉公也说了,可以直接搜索0x080这个常量,寻找符合上面描述的形式的命令.一般来说符合条件的命令也就一两个.跟过去看看是不是有看似LOGFONT初始化的一整块代码.有的话,把前后相关的一小块代码的机器码记下来.不记地址而记机器码是因为用OllyDbg直接加载DLL文件的话,地址转换比较...怪,不像一般的exe文件那样.

然后用16进制编辑器打开DLL文件,搜索并修改,把参数中的80改为86,使游戏可以从对应Shift-JIS改为对应GBK.这里我们在system.dll里搜索C7401490010000C6401B80C6401C02,确定到位于0x00057E41的80需要改为86.

顺带一提,那phantom.exe是有SafeDisc 3系列的壳吧...
