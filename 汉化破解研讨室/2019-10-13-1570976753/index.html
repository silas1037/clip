图片，囧，只能用外置图片

32位与24位图像除了没有透明通道外，在解码上也有一些小差异，等测试通过在补上24位图像的源代码吧
文件大体由五部分组成：文件头、加密的DC直流段、加密的叶节点段、数据块偏移段、数据块段 。
文件头与以前格式有两处差别，一是最后的类型为2，一是被压缩的零字节数为0
以下代码均为C++
Copy code

{
public: char Mark[16] ;           /*====   CompressedBG__   =====*/

 c = key * 346 + b ; c += ( a >>16 ) & 0xffff  ; 
key = ( c<<16 ) + ( a & 0xffff ) + 1 ;

接下来是加密的叶节点，解密算法
有两棵二叉树，第一棵大小为0x10，第二棵大小为0xB0，所以计算也须分为两次

for ( bit = 0 ; result < 0 ; bit += 7 )    //当前字节的高位为1则继续

 result = *( Data + deep ) ;
 *Weight |= ( result & 0x7F ) <<  bit ; 
deep++ ;                        //已经处理的字节数 }


保存好已经使用的字节数，数据块段的起始=加密叶节点段起始+使用的字节数
接下来需要重建二叉树，重建的过程有点奇怪
Copy code
for ( Level = 0 ; Level < Count ; Level++ )
  {
      max = Count + Level ;
      SumNote = 0 ;

              /* 默认的最小节点从第一个有效的节点开始 */

             for ( i = LR + 1 ; i < max + LR + 1 ; i++ )          //左节点以第一个有效节点寻找，右节点以第二个有效节点起寻找
             {

             NewNote = Note + LeastNoteIndex[ LR ] ;

             NewNote->FatherNoteIndex = max ;

      }
      NewNote = Note + max ;
      NewNote->Valid = 1 ;

      NewNote->RightChildIndex = LeastNoteIndex[ 1 ] ;

译码的过程是每bit判断，0走左节点，1走右节点，到达叶节点后返回叶节点的索引。之后从读取后面的叶节点索引个bit作为值，此外还有判断高位等过程，详细情况参见源代码中的处理过程，实在是...
译码后得到是一张N个8*8的 DCT表，需要进行DCT变换。
关于DCT变换参见jpeglib中的DCT解码变换，源代码中的相关部分也是从jpeglib源代码中抄的。
DCT解码之后得到的是Y_Cb_Cr 表，需要进行量化处理。
量化公式：
Copy code
R =  Y + 1.402f * Cr - 178.956f ;
G = Y + 44.04992f  - 0.34414f * Cb + 91.90992f - 0.71414f * Cr  ;
B =  Y + 1.772f * Cb - 226.316f ;


量化之后还需进行边界检查，以去除噪点。
