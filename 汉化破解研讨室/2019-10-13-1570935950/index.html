由于本人没有发帖的权限，所以借用一下风水宝地。请教一个汉化技术的问题。

最近，我正在分析一个游戏的脚本格式。
文件头部分的截图如下所示：



这个脚本的扩展名是.HGO

据我观察，总结为一下几点:

1、 前四个字节，即图中黄色标亮的部分是脚本的区段数。
我们看到，这个脚本里有7个段。我们可以数一下，文件头中，$开头的段正好有7个。这点应该没什么问题。

2、 接下来，我用绿色标亮的部分，共28个字节是给第一个区段名用的。虽然用28个字节给区段名使用似乎不太合乎一般的逻辑(一般应该是16个或者32个)，但是，我观察了好几

个脚本，都未发现其中有其他数据内容。

3、 用天蓝标亮的四个字节，我还不清楚有什么作用，我观察的结果是，只出现过1，2，3三个数字。具体什么作用，目前我还不太清楚，请大家考虑一下。

4、最后用粉红标亮的四个字节，猜测是区段首地址的offset. 具体来说，应该是关于文件头的相对偏移量。所以第一个区段的偏移量是0，其他的依次递增。

接下来就是第二个区段的信息了。

这样，每个区段的信息占用了36个字节。
所以，文件头的大小 = 区段数*36 + 4
对于这个脚本，文件头的大小= 0x100 即 十进制的256 (正好是截图中的最后一行)

以上是本人对这个.HGO脚本格式的一些想法和分析。

表面上看，应该是很简单的一种格式。
具体的文本部分，无压缩也无加密。按理来说，不应该会有什么问题才对。

所以，我很快的编写好文本的提取和替代程序。但是，测试后却发现问题好像还没有完全搞清楚的样子？！

文字部分的提取就不说了，很简单的。关键是文本的回写上！
我按上面的分析写的程序，计算并修正了回写后各区段的偏移量(即粉红色部分的数值)，打包测试运行，却发现运行不太正常！！！ (已经检验了程序的计算上没有问题)

刚开始我猜测可能和文字的偏移有关。
像EGO的第三代scr脚本格式中，专门定义了一个文字的偏移表在文件结尾处。可是，在这种脚本中，我并未发现类的偏移表。在每段文字的开头处，也并未发现类似偏移定位的数

据。所以，对于文字偏移引起的问题我持怀疑的态度。

我估计很可能是那不知道什么作用的四个字节(图上天蓝色部分)出的问题。
有哪位分析过类似的文件，或知道怎么回事儿的，还请帮忙解答，指点迷津。不胜感激。

如果变长不行 意味着脚本内部存在引用其他数据的长度或者偏移的字段 这些字段需要修正。
我的经验是很可能是偏移 而且是发生在分支的时候，比如：

if (条件满足) {
  分支1
  跳转到出口位置
} else {
  分支2
}
出口位置：
......

这里分支1结束部分必须用一个跳转转向出口位置,而出口位置就是一个偏移值（也可以是分支2代码的长度），我想出问题的可能就是类似这样的东西。
