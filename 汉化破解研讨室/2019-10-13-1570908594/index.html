 ONE ～輝く季節へ FullVoice 汉化实战篇[完]

管理提醒：
本帖被 ayazokkon 执行提前操作(2007-06-09)
本来想先写一篇基础理论篇的，结果正式开始写后发现有太多要说明的地方，都不知道从何说起，最后还是决定从实战中积累经验

GAL游戏的汉化从程序方面来说，主要有以下几方面
1、资源的提取（包括解包和格式的转换）
2、脚本修改
3、修改后资源的打包
4、程序的修改（主要是修改成支持中文语系及汉化程序中的日文）

下面用ONE ～輝く季節へ FullVoice简单说明一下其过程

1、资源提取
首先，根据游戏中各文件的名称分析各文件的作用和要提取资源的位置
ONE FullVoice 的文件不多，除了明显的背景音乐文件夹和sav文件夹及可执行程序外，只有以下四个文件
SoundPackSEVo （191M）
SoundPackSEVo.lst （585K）
one （120M）
one.lst （65.4K）
很明显可以看出，游戏中的资源都存在于SoundPackSEVo和one两个文件中，而且从SoundPackSEVo的文件名可以看出里面包含的是SE、Voice等声音。SoundPackSEVo.lst和one.lst的作用值得怀疑。
用winHex开SoundPackSEVo，如下图：




可以看到明显的Oggs开头，说明包内声音是用ogg格式的文件，而且ogg格式的文件没有加密或压缩。但SoundPackSEVo的开头已经是文件数据部分，文件尾部也无特别之处，并未找到明显的文件头和文件索引。再打开SoundPackSEVo.lst（如下图），可以看到里面很结构化，再从文件扩展名lst（list）可以推断，文件索引应该在这个文件中。




注：一般的文件包主要包含文件头、文件索引、文件数据三部分。文件头主要记录文件标记、大小、包内文件数等信息，文件索引主要记录包内每一个文件在包内的位置及数据的长度等信息。文件索引一般在文件包的开头接在文件头后面或者文件包的尾部，像One_FV这样另外用一个文件记录较少见。


再来就要分析文件索引的结构。先要确定文件索引是否压缩或加密。从上图可以看出，文件内有大量重复的00，可以推断文件索引没经过压缩，但是从文件索引中没有看到任何有用的信息，说明索引经过了加密。
文件索引的结构可分为定长（每个文件项占用的字节固定）和可变长。




如上图，根据结构的相似性，从前一个05到下一个05前，可分为一个文件项，可以看出每个文件项固定占用0x4C字节。再找出每个文件项的开始及结束之处。从文件的尾部可以看出，最后是以0x050000结束的。



由此可以推出索引的结构如下




由于不能从加密的索引看出文件包实际的结构，所以要通过调试程序来分析。用Ollydbg调试游戏程序one.exe并在调用ReadFile处下断点，分析读取one.lst或soundPackSEVo.lst后的数据处理（具体的操作不作说明），可以得出lst文件的解密方法如下：
文件头四个字节与0x01010101作异或操作(xor)
每一文件项：
开始两个四字节数分别xor 0x01010101
接着的0x40个字节每字节xor 0x01直至00的字节结束
最后四字节不变
最后得到解密后的索引如下，并可推断出索引的实际结构：




再根据文件索引从文件包SoundPackSEVo中提取第一个文件成功，证明结构分析是正确的。

注：在非压缩的加密算法中，由于异或算法简单高效而被广泛使用。如 A xor B=C，加密后的C再一次异或 C xor B =A 得到原文。xor算法可以是xor某一固定的字节，如Fate的.xp3；或循环xor某一字串，如Comic Party DCE的ar1格式。话说回来，叫One就用01来异或，究竟是有心还是无意呢？

但是，从解密后的one.lst和SoundPackSEVo.lst中都未发现文件的扩展名，而且可以肯定文件包中一定有图片和脚本等文件而不单是ogg的文件。那程序是怎样区分文件的格式呢？从上面的分析可知，还有一个类似标记的部分作用未明，是否用来区别不同的文件格式呢？分析one和soundPackSEVo可知，soundPackSEVo中这个标记全部是05，而one中有01和03的，可以推测程序根据不同的标记区分不同的格式，而05代表ogg。
由于推测是用程序根据不同的标记区分不同的格式，所以再用Ollydbg调试程序，搜索所有的参考文本字符串，在得出的结果中查找ogg，可以看到以下的字符串，而".OGG"刚好是文件后缀格式字符串的第五个。




由此可以推测标记的数值分别代表什么扩展名。根据推导出的算法及逆算法分别可以编写出解包和打包的工具。从实际解包的结果证明，分析的结果是正确的，资源提取成功完成

2、脚本修改

解出One的所有文件包后，我们可发现共有.SNX、.PNG、.OGG三种格式的文件。
其中.PNG和.OGG是我们常见的图片和声音文件，也可以正常打开，并没有特别之处。
可估计脚本文件是.SNX的文件。用WinHex随便打开一个SNX文件，例如AK21.SNK，可发现文件大致可分为两部分，如下图：





但是，并没有发现脚本的文本。

由第一部分大量重复的02可以看出，脚本并没有经过压缩，但却找不到文本，说明脚本经过了加密。
仔细看脚本的第一部分，可以看到最明显的地方是02物别地多，一般来说，脚本如果有大量的重复字符的话，最多的应该是00。从之前文件索引的加密方法，我们可以猜测文件是经过xor 02加密的。按Ctrl+A全选，在编辑菜单中选择修改数据，在弹出的对话框中选择xor，数值填入02，点击OK确认。如下图：



经过变换后，原来的两部分变成如下：





可以看出，第二部分就是Shift-JIS编码的日文文本。分析文本的结构，可以简单的看出，文本的结构为：文本长度（四字节）+文本。汉化这个脚本就是要把日文的文本替换成中文的翻译文本，同时把文本的长度换成翻译后的实际长度。

由于文本是附加在脚本的最后部分，修改后文本长度的改变对整个脚本文件的结构没有太大的影响，所以可以根据需要增加或减少文本的长度。但是如果文本是嵌入在脚本代码中，由于大部分脚本都涉及到跳转等操作，修改文本长度后会引起代码所在地址的变化，所以必须同时更新所有跳转指针指向修改后同一代码的新地址，否则会有意想不到的错误。但由于要找到脚本跳转的代码并作出修改需要对脚本代码有很深的了解，这就要对脚本作深入的分析，了解各代码代表的意义。一个折衷的方法是不改变文本的长度，如果翻译后的文本比原文短后面就用00（ASCII码）填充，但翻译后的文本不能超过原文的长度。

最后，虽然脚本文件的文件头往往会有一些有关脚本文件的信息，如文件大小、代码段长度、文本开始的地址、文本段长度等信息，我们要对有改变的作出相应的更改。
先用一些常用的信息与文件开头部分作对比：
文件大小：0x099C
代码段长度：约 0x07B8 （部分可能是文件头部分）
文本开始地址：0x07B8
文本段长度 ： 0x01E4



对比可知，第一个4字节数0xA4暂时未有对应，第二个4字节数 01E4 对应文本段长度。由于修改后的文本长度有变，所以这一个数要作出相应的修改。后面的数无特别之处，大概是脚本的代码了
分析第一个数0xA4，先看0xA4对应地址的代码，并无特别之处，与文件大小、代码段大小也相关很多。可以考虑是否与某个数存在倍数的关系。以0x07B8除以0xA4，得到约0x0C，0x0C乘以0xA4得0x07B0。考虑到头8字节可能是文件头，这一关系可能成立。再用其它几个脚本文件校验，证明这一关系成立。可以猜测这个数为脚本的代码数，每一代码固定占用0x0C个字节。这一个并不需要修改，但对编程确定文本段开始的位置有重要的作用。

最后，再用上面的方法与02作xor重新加密脚本，即可以得出修改后的脚本文件。

4、程序的修改

程序的修改主要是修改成支持中文语系及汉化程序中的日文。程序中字符串的存在方式与脚本中文字的存在方式大致相同，可分为两种方式，一是以资源的方法存在于可执行文件的尾部，这种方式的文字要修改很简单，只要用资源修改软件，如 PE Explorer、Resource Hacker等软件就可以轻松修改，对文本的长度也没有限制。另外一种是存在于可执行文件内部的文本，同样与上面所说的脚本修改方法类似，长度比原文短的用00填充，但不可以超出原文的长度。程序的汉化大致上就是查找替换等工作，这里不作详细讲解了。
下面说说程序支持语言的修改
目前很多日文GAL在XP系统下都可以不用任何转换内码的软件如AppLocale就可以正常地显示出日文字，这对游戏玩家无疑是很大的方便，但对游戏汉化又是一大麻烦。因为当用支持shift-JIS编码的程序显示翻译后的GB2312的脚本，出来的将会是一堆乱码

要修改程序使原程序从支持shift-JIS编码变为支持GB2312编码，通常我们要修改游戏中调用的系统函数CreateFont或CreateFontIndirect所调用的参数中的字符集的值

从MSDN Library中我们可以查到到CreateFont或CreateFontIndirect的函数原形

HFONT CreateFont(
int nHeight,           // height of font
int nWidth,           // average character width
int nEscapement,       // angle of escapement
int nOrientation,       // base-line orientation angle
int fnWeight,         // font weight
DWORD fdwItalic,       // italic attribute option
DWORD fdwUnderline,     // underline attribute option
DWORD fdwStrikeOut,     // strikeout attribute option
DWORD fdwCharSet,       // character set identifier
DWORD fdwOutputPrecision, // output precision
DWORD fdwClipPrecision,   // clipping precision
DWORD fdwQuality,       // output quality
DWORD fdwPitchAndFamily,   // pitch and family
LPCTSTR lpszFace       // typeface name
);

HFONT CreateFontIndirect(
CONST LOGFONT* lplf   // characteristics
);
其中 LOGFONT的声明如下：

typedef struct tagLOGFONT {
LONG lfHeight;
LONG lfWidth;
LONG lfEscapement;
LONG lfOrientation;
LONG lfWeight;
BYTE lfItalic;
BYTE lfUnderline;
BYTE lfStrikeOut;
BYTE lfCharSet;
BYTE lfOutPrecision;
BYTE lfClipPrecision;
BYTE lfQuality;
BYTE lfPitchAndFamily;
TCHAR lfFaceName[LF_FACESIZE];
} LOGFONT, *PLOGFONT;

CreateFont和CreateFontIndirect的功能是完全一样的，只是CreateFontIndirect使用了一个LOGFONT结构的参数包括了CreateFont中的所有参数
要改变程序支持的字符集，就要改变程序调用上面两个函数时的fdwCharSet或lfCharSet的值。其中各字符集所对应的值如下：

一般日文游戏汉化经常用到的字符集有SHIFTJI_CHARSET(日文) 、GB2312_CHARSET(简中)、CHINESEBIG5_CHARSET(繁中)所对应的值分别为128、134、136，转换成十六进制为0x80、0x86、0x88。简单地说，要使程序能正常使用中文，就是要把程序中字符集的值由80改成86(简中)或88(繁中)

下面以ONE ～輝く季節へ FullVoice版实际说明一下具体的操作：
首先，先确定程序是否需要修改，直接运程游戏程序，可以看到程序在没有使用AppLocale的情况下可以正常显示日文，说明程序有需要修改。



用OllyDbg载入one.exe并运行，搜索当前模块中的名称



在弹出的窗口中可以找到CreateFontIndirectA，右击函数的位置，在弹出菜单中选择在每个参考上设置断点



然后转到游戏窗口进入新游戏。程序很快就在调用CreateFontIndirectA的地方停下来。如下图：



其中EBX(00E4DCDC)的值为CreateFontIndirectA的参数，在数据窗口中跟随EBX，可以看到如上图数据窗口中是一个LOGFONT的结构，红线圈住的80就是字符集的值，后面的“俵俽 僑僔僢僋”是字体名称，即shift-JIS编码的“ＭＳ ゴシック”，就如简体中文的宋体一样，是日文系统的默认字体，一般在日文的程序中我们经常可以看到。
将数据窗口中的80改成86，按F9运行程序，程序再次在同一位置中断，查看EBX指向的值发现，字符集的值已经为86,说明字符集没有重新赋值，取消此处的断点，再按F9运行程序，可以看到程序中显示的已经变成了乱码，也就是说修改刚才找到的字符集的值可以成功地把程序变为支持GB2312编码了。



接下来就要找到给字符集赋值为80的地方，将80改成86。先从中断的位置往上看，看附近有没有给字符集位置(EBX+17)赋值的语句，但没有找到，所以要想另外的方法去找。考虑到一般编程的习惯，给一个结构(这里是LOGFONT)的各数据赋值往往是写在同一块的，也就是说，在给LOGFONT的字体名称赋值为“俵俽 僑僔僢僋”的附近，极有可能有给字符集赋值为80的代码。

由于OllyDbg对非ASCII码的字符串支持不好，所以要先用ASCII码的字符替换“俵俽 僑僔僢僋”。先退出OllyDbg，再用WinHex打开one.exe，查找“俵俽 僑僔僢僋”，再用任意一特殊的ASCII码字符串替换，例如我用了"holand000000"，保存并关闭WinHex。再用OllyDbg载入one.exe，搜索所有的参考字符串，在弹出的窗口可以找到"holand000000",如下图



双击"holand000000"所在的行，自动跳到调用调用"holand000000"字符串的地方



仔细观察附近的代码，可以发现有"mov byte ptr ds:[ebx+17],80"这一段代码，这极有可能就是我们要找的地方，双击此处代码，在弹出的“汇编”窗口中将80改成86并点击“汇编”按钮。按F9运行程序，进入游戏后就可以发现确实能达到目的，说明我们要修改的代码就是此处。

记下要修改处的几行机器代码 885315885316C6431780，退出OllyDbg，再用WinHex打开one.exe，查找16进制值“885315885316C6431780”，查找到结果后再按一次F3查找下一个显示没找到，证明查找到的是唯一的，再将最后的80改在86，即“885315885316C6431786”,然后保存退出，这样修改基本上就完成了。（也可以不用WinHex修改，直接用OllyDbg里的OllyDump直接将修改后的程序直接保存下来）

如果游戏有选择字体的功能的话，还要修改使程序可以选择中文的字体，一般是列举字体用到的函数为EnumFontFamilies 、EnumFontFamiliesEx、EnumFonts，函数原形可参考MSDN，里面有详细的介绍。修改的原理大致同上面所说的相同，这里就不作详细介绍了。

One,特别是老的One的资源归档还是相当trivial的,相当适合教学嗯.期待大作的完整放出.

对于固定串的xor操作,在分析得到那个固定串的值之后,可以使用XorIt来方便的解决.当然,这东西是收费的.破解过的XorIt很多地方都能找到. http://sinnercomputing.com/XorIt.htm
对于循环xor,还是自己写个小程序解决比较方便,虽然不一定多快.
使用固定串或循环xor是最常见,最容易实现也相对容易解决的非压缩加密方式.为了实现运行时的解密,用于xor的两个源必须同时存在,其中一个是待加密的字节,另一个是用于加密的字节(固定或循环).跟踪调试很容易找到用于加密的字节(要么内嵌在程序里,要么寻找外部的加密用文件),因而可以顺利解决.

对于获取程序内引用过的字符串,使用URSoft的W32Dasm也很方便的.嗯,不过具体调试还是习惯用OllyDbg.
这两个工具的具体使用方法恐怕属于工具的基础知识掌握,在专门的破解教学论坛,例如看雪那边有相当详细的教程.
http://www.pediy.com

楼上说到AFS开头的话,多半是PS2和DC上使用的归档格式吧.开头的4个字节是0x41 0x46 0x53 0x00.
既然是归档格式,里面到底装着什么都有可能.不想多麻烦的话,可以使用AFSExplorer来打开查看里面到底放了些什么文件,解包和重新打包也可以通过这个工具完成.
