 Eushully社ARCGameEngine中文化支持简记，以《魔導巧殻体験版》为例

ARCGameEngine（下文简称AGE）中文化流程：
【资源导出
解包DATA.ALF（如果有资料片还需解包APPEND.ALF）
|→得到BIN二进制脚本文件
|→得到（AGF）BMP图片资源文件
】

【资源修改
《图片
|→新版AGE可以直接读取32位BMP，无需回封成AGF文件，直接将位图拓展名改为AGF即可（感谢vonjack大大提供信息）

》

《脚本
BIN脚本由
HeadInfo(60字节，此部分数据对引擎无影响)
OP codes(不定长)
经过取反的文本(不定长，可有可无)
ExtraData01(不定长，可有可无，起始偏移标记于HeadInfo第40字节)
ExtraData02(不定长，可有可无，起始偏移标记于HeadInfo第48字节)
ExtraData03(不定长，可有可无，起始偏移标记于HeadInfo第56字节)


本游戏从0开始到1023共有1024个op code，有效op code为476个。每个op code的含参数长度记录AGE.EXE内。

简单列举几个涉及文本的op codes
Copy code
op code[hex]    Description
      6E    对话文本
     140    系统
     192    系统、战斗、人名、道具等等
     194    表示项目
     195    系统
     196    文字注音
     1A7    系统
     204    表示项目


需要注意的并不是说出现了以上op就一定会对应文本，op可以不对应任何文本、或者指向空字串、
或者指向前面已经出现过的句子，还可以指向前面已经出现过的句子的一部分。

比如    
    魔导巧壳手办战争
    0 1 2 3 4 5 6 7

前面已经有op指向过这句，后面的op也一样可以指向这句，op还可以指向这句的任意位置，比如4位置（手办战争）
，回导BIN文件的时候要注意这点的处理，不然可能会出现一句话无限循环显示的问题。

除了op＝196时前后2个offset都是文本外，其他op都是靠后面的那个为文本，前面的为项目ID，另外op＝55的不用改。

同时若修改后的文件长度发生改变，还需要修改op参数里面对应ExtraData的地址，不然AGE会挂的。
》
】

【EXE修改思路
|→EXE脱壳                                           |→完成
            |→边界检查→字体问题导致的VC＋＋弹窗报错
|→EXE不脱壳                                         |→内存补丁＋内联钩子→完成
】

【在EXE里找op code长度地址
从解包的文件夹里找一个*INIT.BIN的文件扔到游戏目录下，
OD加载EXE，bp CreateFileW，F9两次后age.exe主程序就已经还原了，一直F9到读*INIT.BIN的时候，
bp ReadFile，等它读取长度大于60时按Ctrl＋F9，然后F8单步返回程序领空继续单步，然后如图：



红色箭头那里就是op code长度结尾部分，把滚动条往上拉，会看见一坨很有规律的int值，
如果突然有个地方的地址跟别的不同，那个地方下一个int就是整个op code长度集合开始的地方。
然后把集合保存下来，将每个op code的长度值从exe里面找出来吧。

幻燐2有512个op（2048字节），魔导有1024个op（4096字节）。

将op code长度值写出来就是个比较费人工的操作了，因为得一个个手动来＝＝

这个集合的op code是按照从0——1023的顺序排列的，集合里的地址就是每个op code在BIN文件里的含参数长度。

用hex编辑器打开保存出来的集合文件，每个op对应的长度地址是op×4，
那么op＝1的时候，它的长度值在集合文件的地址就是1×4＝4，



我们可以看到地址是418460，进入OD按Ctrl＋G，输入418460，当你见到这样的指令

mov dword ptr ds:[ecx+edx*8+5D4C4],1

时，那个1就是当op＝1时的含参数长度（1×4＝4字节），因为本身op code就有4字节，所以这个op是无参数的。

E社游戏给oplen赋值都是mov dword ptr ds:[ecx+edx*8+XXXXX]这样的形式。

再比如op＝6对应的地址是41BB00，进入exe后会发现 mov dword ptr ds:[esi+ecx*8+5D4C4],5 这句
那么5（5×4＝20字节）就是op＝6的含参数长度。

下面再来说说op＝0时的参数长度地址，你会发现有很多op都是这个地址，而这个地址在exe是没有一个明确告诉
到底这个op的长度是多少，我的做法就是当扫描脚本的时候如果遇到这个op，程序自动停下然后我根据前后参数
人工补上那个op长度值，这类地址的op在游戏里绝大部分都未用到（本游戏中此地址的op全部未使用），
所以其实要手动补充的并不多。
】

【字符边界检测＆弹窗报错】
《边界检查
在exe里搜索常量7F，然后你见到

XXXXX  cmp xxx,20 xxxx
……
XXXXX  cmp xxx,7F xxxx
……
XXXXX  cmp xxx,3F xxxx

类似这样的地方，就是边界检查了，地方挺多的，将每个地方的 3F 改成 00 ，这个地方
的上面如果有类似xxx-A0或者xxx+60，将－A0或者＋60一并改掉，避免A0打头的GBK双字节字符被当成单字节字符，
这样边界检测就被艹掉了。

然后搜索常量，把8140 换成 A1A1 ，这个修改是避免GBK的全角空格显示成口。
》

《弹窗报错
搜索字符串“寈崘丗[%s]”，找到EXE对这个字符串的引用，然后在它上面一个call下断，运行的时候就会发现
日文字体名被作为参数传入


跟进断点那个call把字体名比较的循环跳转改掉就可以了。
》
