 FFDSystem资源破解的讨论

朋友打算汉化little witch的游戏，所以帮忙分析了一下，但是遇到一个问题，所以发到这里来以求群策群力，解决问题。
测试游戏：q？？？！se 体验版（我根本没努力的去记住它。。orz）
注意：整个文章只有整个解密算法（除了ver5的分支）是正确的，其他的信息并没有得到充分验证。
md5算法的实现细节，网上有很多，我用的是openssl-0.9.6a的。

这个游戏的封包是.dat扩展名，加密算法是md5，压缩算法lzss。
索引段部分是一个自定义的混淆算法。
这里的关键是资源的加密处理方式。
游戏首先从system.dat读取一些基本的资源数据: *.def
然后用资源文件的名字（比如a.def）进行md5，然后利用" "进行md5的迭代运算，最后再对数据进行解密。

这里的关键就是要解出数据，就必须有资源文件的名字。而游戏的索引段存储的是资源名的md5，所以问题就变成了寻找资源文件名字了。

游戏没有在某个特定文件或者特定封包内的特定资源文件内存放资源名称列表，而是完全按照游戏运行时脚本内的资源调用动态获取资源的，所以试图按照常规的提取方法是不行的。

仔细观察了一下解密算法，可以发现它的md5迭代部分首先是以资源名称md5后的hash开始的，所以这里用了md5劫持和重放的手段，下面的代码里#if 1的分支是原本的解密算法，而#if 0就是用了md5劫持和重放，将第一次的hash结果和现场数据拷贝到ctx中，以重现原本第一次hash运算后的正确结果。（name参数的作用原本是要显示的提交一个明文资源名称的）但是问题并没有这么简单，因为解密算法是md5迭代，所以2次md5之间有部分数据是关联，这个关联数据之一（时间仓促，我现在100％确认的就是我现在要说的这1个字段）上一次md5要hash的数据的长度，也就是资源文件名的长度。

不是真正实用化的参考代码：（Nl=0x90是我测试某个特定的资源文件时候用的，那个资源文件名长是18字节，这里要x8,得到0x90)
Copy code
static int FFDSystem_parse_system(const char *archive, const char *name)
{
    TCHAR path[MAX_PATH];
    HANDLE fd;
    dat_header_t dat_header;
    dat_entry_t *index_buffer;
    unsigned int index_entries;
    unsigned int index_buffer_length;
    char key[MAX_PATH];
    char *dat_name;
    BYTE md5[16];
    dat_entry_t dat_entry;
    BYTE *compr;
    unsigned char dec[64][16];
    char seed = ' ';
    MD5_CTX ctx;
    int key_len;

    if (acp2unicode(archive, -1, path, SOT(path)))
        return -1;
    
    fd = MyOpenFile(path);
    if (fd == NULL)
        return -CUI_EOPEN;

    if (MyReadFile(fd, &dat_header, sizeof(dat_header))) {
        MyCloseFile(fd);
        return -CUI_EREAD;
    }

    if (strncmp(dat_header.magic, "RepiPack", 8)) {
        MyCloseFile(fd);
        return -CUI_EMATCH;    
    }

    if (dat_header.version != 4 && dat_header.version != 5) {
        MyCloseFile(fd);
        return -CUI_EMATCH;    
    }

    dat_name = (char *)malloc(dat_header.dat_name_length + 1);
    if (!dat_name) {
        MyCloseFile(fd);
        return -CUI_EMATCH;    
    }

    if (MyReadFile(fd, dat_name, dat_header.dat_name_length)) {
        free(dat_name);
        MyCloseFile(fd);
        return -CUI_EREAD;
    }
    dat_name[dat_header.dat_name_length] = 0;

    if (dat_header.version == 4)
        decode((u32 *)dat_name, dat_header.dat_name_length, 0xb76522ab, 0x38f5cb86);
    else if (dat_header.version == 5)
        decode((u32 *)dat_name, dat_header.dat_name_length, 0x8AF4622E, 0xCE4985A0);

    if (lstrcmpiA(dat_name, PathFindFileNameA(archive))) {
        free(dat_name);
        MyCloseFile(fd);
        return -CUI_EREAD;
    }
    free(dat_name);

    if (MyReadFile(fd, &index_entries, 4)) {
        MyCloseFile(fd);
        return -CUI_EREAD;
    }

    index_buffer_length = index_entries * sizeof(dat_entry_t);
    index_buffer = (dat_entry_t *)malloc(index_buffer_length);
    if (!index_buffer) {
        MyCloseFile(fd);
        return -CUI_EMEM;
    }

    if (MyReadFile(fd, index_buffer, index_buffer_length)) {
        free(index_buffer);
        MyCloseFile(fd);
        return -CUI_EREAD;
    }

    key_len = strlen(name);
    for (int n = 0; n < key_len; n++)
        key[n] = tolower(name[n]);
    MD5((const unsigned char *)key, key_len, md5);
    for (unsigned int i = 0; i < index_entries; i++) {
        if (dat_header.version == 4)
            decode((u32 *)&index_buffer[i], sizeof(dat_entry_t), 0xF5FACF01, 0x38F5CB86);
        else if (dat_header.version == 5)
            decode((u32 *)&index_buffer[i], sizeof(dat_entry_t), 0x1AF56CDB, 0xCE4985A0);
        if (!memcmp(md5, index_buffer[i].name_md5, 16))
            break;
    }
    if (i == index_entries) {
        printf("not found\n");
        free(index_buffer);
        MyCloseFile(fd);
        return -CUI_EMATCH;
    }
    memcpy(&dat_entry, &index_buffer[i], sizeof(dat_entry));
    free(index_buffer);

    if (MySetFilePosition(fd, dat_entry.offset, 0, IO_SEEK_SET)) {
        MyCloseFile(fd);
        return -CUI_ESEEK;
    }

    compr = (BYTE *)malloc(dat_entry.comprlen);
    if (!compr) {
        MyCloseFile(fd);
        return -CUI_EMEM;
    }

    if (MyReadFile(fd, compr, dat_entry.comprlen)) {
        free(compr);
        MyCloseFile(fd);
        return -CUI_EREAD;
    }
    MyCloseFile(fd);

    MD5_Init(&ctx);
#if 0
    memcpy(&ctx, md5, 16);    // A,B,C,D
    ctx.Nl = 0x90;
//    ctx.data[14] = ctx.Nl;
#else
    MD5_Update(&ctx, key, key_len);
    MD5_Final(dec[0], &ctx);
#endif

    for (i = 0; i < 64; i++) {
        MD5_Update(&ctx, &seed, 1);
        MD5_Final(dec[i], &ctx);
    }

    BYTE *act_data;
    DWORD act_data_len;
    decode_resource(compr, dat_entry.comprlen, 0, &dec[0][0], 1024);
    if (dat_entry.uncomprlen == dat_entry.comprlen) {        
        act_data = compr;
        act_data_len = dat_entry.comprlen;
    } else {
        BYTE *uncompr = (BYTE *)malloc(dat_entry.uncomprlen);
        if (!uncompr) {
            free(compr);
            return -CUI_EMEM;
        }
        lzss_decompress(uncompr, dat_entry.uncomprlen, compr, dat_entry.comprlen);
        free(compr);
        act_data = uncompr;
        act_data_len = dat_entry.uncomprlen;
    }
    free(act_data);
    return 0;
}


既然没有办法拿到资源文件名，那么资源文件名长度自然也不得而知了，所以这里改为暴力破解。
因为ffd的资源文件总是有3字节的后缀的（这是我的假设），所以文件名至少5字节，假设最大长度是32字节，那么这里用一个循环来解密：如果解密和压缩出来的的资源数据正确，就退出循环，否则文件名长度计数＋1，重新解密。

这里有个问题：数据正确的依据是什么？

对于压缩了的数据，这个很好判断，只要解密后压缩的数据，其实际解压的长度==索引段中记录的解压长度即表示数据解压正确。
但实际上要知道我们不知道之前解密后的数据是否正确，所以解压时候有可能是一堆错误数据，所以应该在lzss解压代码中加入当前压缩数据的位置和整个压缩数据长度的边界检查，1是防止程序运行错误（访问非法地址），2是这个条件也能成为解压完成的一个标志－当（当前的压缩数据的位置＝＝整个压缩数据的长度 && 实际解压的长度==索引段中记录的解压长度）时，解压完成，且解压出的数据一定正确，证明之前的解密也是正确的。

对于没有压缩的数据，就有点麻烦了。因为不存在解压算法那种“结束条件”，所以判断解密是否正确地依据就是直接判断解密后的数据了。统计一下，ffd的资源数据包括ogg、png、def、tkn（还有别的什么我暂时不知道）。这些数据除了def，都有明显的magic；def是sj编码的纯文本文件（用IsDBCSLeadByteEx(932,  TestChar)逐个字节检查咯，不，也许用挂MB_ERR_INVALID_CHARS参数的M2W更好） 。所以对于未压缩的数据，解密后依次和各个类型的资源进行比较，如果符合其中之一，就表示解密成功，退出解密循环。当然压缩的数据也要简单判断一下，以便保存文件时添加适当的后缀名。

至于最后保存的资源文件，只能以md5的字符串来命名了。

大家有什么好的办法欢迎提出，比如说真的有资源文件名列表之类的。
FFDSystem的cui插件目前制作中，差的就是资源类型的匹配上了。

最后这游戏真汉化的化，分析tkn二进制脚本就是必然的了，我现在实在不想干这事了，有人报名吗？
