FAVORITE引擎VM及脚本结构分析

本文中提到的地址（主要在vm中）以色鸟鸟（いろとりどりのセカイ）为例。

一 总体结构


虚拟机脚本就是.hcb文件了。
注意引擎在启动的时候是以找到的第一个.hcb文件为准，所以拿色鸟鸟的world.hcb来说，你建立一个a.hcb放进去前者就被忽略了。
虚拟机代码全部在exe里面。
hcb结构比较简单，前面是操作码+操作数，最后是一个类似PE文件输入表的东西。

二 VM


先说vm本身吧，最值得关注的是ScriptObject对象。443507处的代码
00443507  .  8B0495 A0F547>MOV EAX,DWORDPTR [EDX*4+47F5A0]
执行到这里时，esi指向的地址即为该对象。
分析出该对象及相关结构结构如下：
#pragma pack(2)
//栈entry
struct StackEntry{
         u8type; //0--bool0 1--bool1 2--int 3--float 4—str 5,6?
         u8type?;
         u16stackBase;
         u32value;
};

struct ScriptObject{
        intunk1;
         LPVOIDscriptBuffer;
         StackEntryst[0x100];
         intunk;
         intunk;
         intbEnd;
         StackEntrytemp;
         intunk;
         intscriptPos;             //当前执行到的脚本位置
         intstackDepth;        //栈顶（相当于esp）
         intcurStackBase; //当前函数的栈底（相当于ebp）
}

虚拟机每次从脚本中读取一个字节作为控制码，然后通过443507的语句获得控制码处理函数并调用。控制码共0x27条。
虚拟机有一些系统函数供脚本调用，脚本中用到的函数都在脚本最后的输入表中记录。如MotionMoveZStop，AudioLoad等。
就这么多吧，VM本身没有做太深入的研究。


三 脚本


1 文件结构
文件的第一个dw是一个文件offset，指向HcbHeader。HcbHeader如下：
struct HcbHeader{
         u32EntryPoint; //代码入口点，这是一个文件偏移
         u16count1;
         u16count2; //这两个count好像被用来分配内存了……具体忘了
         u16resolutionIdx; //exe中的分辨率表的索引，指定游戏的分辨率
         u8len;
         chartitle[len]; //游戏窗口标题
};
Header之后是输入表，第一个u16为输入函数个数，之后每个表项结构如下：
struct ImportEntry{
         u8type?; //忘了分析这个字节……
         u8len;
         charname;
}
文件到此终止。
VM会根据名字自动寻找对应的函数地址，生成一个真正的输入表，脚本中直接通过序号调用这些函数。

从文件的第一个dw之后到Header之前都是字节码段。

2 字节码参考
一个字节为控制码，之后为操作数，操作数长度由控制码决定。每组之间一般没有间隔（理论上可以有，像花指令那样，会对文本提取造成一定的障碍）。以下是码表
op              len            
01              2                InitStack(u8 num,u8 unk);//为一个函数初始化栈空间，push(num<<8)&unk，(push 0)*num
02              4                call(u32 offset); //push(x.stackBase,x.curPos);jmp offset
03              2                callSys(i16 num); //调用系统函数
04              0                ret; //释放栈数据，返回
05              0                ret1; //scriptObject.temp=pop;ret
06              4                jmp(u32 offset);
07              4                jz(u32 offset); //如果栈顶为0则跳转 s--
08              0                push0();
09              0                push1();
0A              4                pushInt32(i32);
0B              2                pushInt16(i16);
0C              1                pushInt8(i8);
0D             4                pushFloat(f32);
0E              ?                pushStr(i8 nLen,char str[nLen]);//push &str
0F              2                pushGlobal(i16 num); //pushGlobal[num]
10              1                pushStack(i8 num); //将当前栈底+num处的栈内容提出来入栈，s++
11              2
12              1
13              0                pushtop(); // push top; s++
14              0                pushtemp(); //pushscriptObject.temp; scriptObject.temp=0; s++
15              2                PopGlobal(i16 num);//Global[num]=pop
16              1                stackcpy(i8 num); //与i10相反，s--
17              2                //s-=2
18              1                //s-=2
19              0                neg()
1A              0                add; //a=pop;b=pop;push a+b
1B              0                sub; //a=pop;b=pop;push a-b
1C              0                mul;
1D             0                div;
1E              0                mod;
1F              0                bitTest; //a=pop;b=pop;pusha&(1<<b)?1:0
20              0                CondAnd;//a=pop.type;b=pop.type;push x.type(1or0)
21              0                CondOr;
22              0                SetE; //a=pop;b=pop;pushx.type(a==b?1:0)
23              0                SetNE;
24              0                SetG;
25              0                SetLE;
26              0                SetL;
27              0                SetGE;

上面的指令说明里面，s表示栈，x表示ScriptObject。
有4个指令意义不明，原因是StackEntry中，除了已知的4种类型外，还有两种类型没有分析出来，这几条指令和它们相关。

3 字节码分析
整个hcb脚本如同一个编译完的pe文件，在编译之前应该也是由一个个源代码文件（函数）组成的，下面说下如何拆分出这些函数。
第一种办法比较暴力，即直接扫描。每个函数起始似乎都有一个InitStack指令，用来初始化栈和进行局部变量的空间分配，而且该指令似乎只出现在函数起始。那么就从代码段首扫描到段尾，一旦发现此此指令就认为遇到了一个新的函数。注意前面用到了两个“似乎”，虽然我拆分色鸟鸟最终就用的这个办法，但是也没办法保证准确性。只能说提出来的文本部分按章节分配正确。
第二种办法是函数跟踪。从入口点开始依次执行代码，一旦遇到call指令，即认为这是新函数起始，用递归遍历的办法将整个代码的所有路径走完，即得到所有函数。本方法的前提是不存在jmp到其他函数体内的语句，这一点一般可以满足，不过除此之外难点在于好像有些用于VM的回调函数……我首先尝试的这个办法，然而只分析出了很少几个函数，又懒得从那成千上万条指令里面找回调函数添加的代码了，就放弃了。

四 汉化相关


最后说点汉化的东西，首先文本提取的话，直接找pushStr指令就行了，人名的话是通过序号显示的，有个专门的函数，跟踪一下找到对应表就OK了。可以按照函数分开。然后是显示，我是写了个d3d9让它读取指定目录下的txt文件在游戏中显示，翻译比较方便。要改的地方包括：
1 hook pushStr指令，让它读取txt的字符串；
2 跳过字符串地址校验；（这引擎会检查push进去的字符串偏移是否在hcb范围之内）
3处理GetGlyphOutline对于全角空格获取不能的问题；（否则是显示不了0xa1a1字符的）
4 处理CompareString函数的locale参数。（二分法查找sjis字符串必须遇到的问题）
5 其余各种本地化，字符集之类的。
这引擎没有字符集检查。
图片啥的就不管了。
