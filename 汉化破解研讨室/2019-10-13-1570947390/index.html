按理说事情到这里应该就结束了，但是发现遗留了2个问题：

问题1. 这2张图，本质上其实是一样的(注*)，都是800 x 600 x 24bits带alpha通道的图。但问题是，我请教的朋友里，多数人都声称用ps打开看，只有一片透明。

到底是什么原因，至今未明。。。。orz
还有就是有个人说用ps可以打开，但是没有看到alpha信息...orz

问题2.我在写decoder的时候，首先把idat数据合并在一起组成zlib压缩数据流，然后解压缩。根据协议定义：实际解压后的字节是：800 x 600 x 32bit / 8 + 600。

这里变成32bit是因为有1字节的alpha信息，排列方式是rgbargbargba.....

多出来的600正好是height个字节，意味着每行rgb数据的最开始多出第一个字节，经过调查，发现这个字节通常填0，但是具体实际意义不是很清楚。但是在解h011.cwp.PNG的时候，我特意检查了每行的第一字节，发现不都是0（准确说没有1个是0）：（这图从左上角往右一行行看，每个数字表示每行第1个字节的值）

每个scanline之前会有1字节的filter type,目前的PNG标准只定义了一种filter method,其中有5种filter type,值范围从0到4.Filter是为了提高压缩率而做的,会改变PNG图象中字节的值,嗯注意是对字节而不是像素的操作.

type 0对应的是无filter...把filter值都改为0会使解码器重建图像的时候进行错误的解析.得到的就是那黑乎乎的图.

具体参考PNG spec, http://www.w3.org/TR/PNG/#9Filters
贴上来会很麻烦所以还是直接去看方便.
