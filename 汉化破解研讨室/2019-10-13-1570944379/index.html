零起点GBA讲义第一章

配置使用eclipse搭建的GBA开发环境


作者：苏联OKB294驻澄空学园捷乌什卡实验所
            陆军少校  沃尔夫苏利文

序言

刚看见这个标题可能就会觉得有点怪怪，现在的环境发展已经远远的超越了GBA盛行的年代。硬件有NDS，PSP甚至PS3，Xbox，软件支援有java，XNA等现成的非常方便的开发包，为什么还要研究GBA的开发呢？这个问题有多方面的考虑角度……就老衲的情况而言，GBA与其说是程序设计，不如说是对于综合硬件基础的一个锻炼和补充。虽然本文的定位十分白，但是为了避免有更白的人出现导致卡住或者不明原因触雷，因此，假定此系列文章的接受对象已经拥有如下基础：

*知道基本的计算机结构和操作 (CPSC101)
*修毕“C语言程序设计”，对C语言有大致了解并知道如何操作文件读取和写入 (APSC160)
*拥有基本的线性代数知识和不是十分差的概念理解和计算能力(MATH223)
*有点基本的硬件电路概念即可
*有点基本的单片机概念即可

那么进行GBA研究的目的呢？我想最主要的有以下几条：
*进行单片机为基础的游戏开发
*更好的理解底层编程的含义（有点类似做库和驱动程序的味道）
*掌握基础图像技术（特别是调色板……有人说它落后实际上变幻无穷）
*最后有能力的当然是自己做一些力所能及的开发了……
*GBA在原理上和NDS一样，适合作为一个简单的的入门基础

现在网上所能找到的教程，由于年久失散，中文教程有据可查的为数不多，主要是水银等针对mode4的初级教程。很多软件，自己写的库都无从寻找，因此在这里也将对前辈们所做的工作作一个整理。还有非常重要的一点，就是针对在学习和钻研的过程中，可能会遇到一些意想不到的问题，有些是概念上的盲区所致。在这点上也将尽力给予提醒和表达。

嘛，在开篇之余，不想重复进行GBA的介绍，因为大家都知道这是个什么东西。至于对重要技术参数的评论，待到正式讲解GBA原理时才进行……图解……

开发GBA的时候，时刻要记住一点，你所面对的不是由一大堆库，以及成型的函数所搭建的软件开发体系，而是直接面对硬件的硬件开发体系。如果通俗一点的话，就是“没有现成函数可用的C“（当然不否认可以直接用汇编来做。可以说整个一套开发原理和单片机的开发是一模一样的（而且还少了不少昂贵的开发板仿真器，麻烦的外围电路，接线云云……）说无论什么样的人，只要有心开发学习，都可以从中找到它的乐趣。因此，请诸君一起加油吧=v=

[赶赴北京飞机早睡，明天补齐后边的内容……]

一。GBA的CPU，编译器，以及基础的devkitadv开发包

GBA的CPU为一颗32位的主频为16.78Mhz的ARMCPU。虽然速度非常可怜，但是配合以其强劲的2D硬件加速机能，以及超小的屏幕，处理一些不算复杂的游戏还是拥有足够的功能的。但是，这样的CPU需要特殊的ARM汇编指令集来进行操作。使用汇编指令来写游戏并不是一个好主意。因为虽然这样做效率最高，但是对于程序工程开发来讲，第一不利于整体工作效率，第二容易出错不易查找，第三……因为这样做相当于用自己人肉做的大脑去充当编译器，所以可能在一些非常细小和机械化的地方消耗精力以及产生不必要的纠缠。既然机器能做的事情为什么不交给机器呢？所以我们首选的语言是C。C既承接了结构化程序设计的优点，又拥有面向机器的功能和效率，最重要是普及率高上手方便。因此在本系列教程中，用C作为贯始至终的工作语言。

谈到C，就要有适合于这颗ARMCPU的编译器。编译器可以把C语言翻译成机器能懂的语言。现在GBA开发有两种编译器可供选择。一种是gcc，虽然效率不算最高，但是有免费的优势，而且为几乎所有的商业游戏公司所采用——就连任天堂开发游戏也用的是gcc。这样，大部分民间函数库都是使用gcc的，devkitadv所带的一个小小的函数库（就是#include里边的那些东西）就是用gcc来写的（其实gcc还有个配置麻烦以及需要写makefile的毛病，但是在我们讲到devkitadv会完全解决这些问题）
还有一个编译器就是传说中的ARMSDT。ARM公司自家开发的编译器，当然效率最高，但是……是要收费的……传说中有地板版本，但是由于其用的人实在是太少了，现在很难碰见。有个不幸的消息……黄炎中前辈所开发的Gslib主要是使用ARMSDT的，当然据说后来也出了Gcc版本，但是四年前的文章保存到现在已属不易，数据就更难说了。因此在选择编译器上，本教程选择的是gcc。在拿到一份新的开发库是，一定要弄清楚是Armsdt版还是gcc版，以免误用造成疑惑和时间的拖延。

以上我讲到了编译器需要配置以适应arm的cpu，以及编译时要写makefile，这些是十分复杂而麻烦的。但是伟大的人民群众研制了一个已经配置好了的套件，并整合了简单的开发环境进去。这个套件的名字就叫devkitadv。

devkitadv下载地址：
http://devkitadv.sourceforge.net/

一路点下去，安装到C盘，因为我们只需要它的编译器，在C盘的话很多东西都是默认的不用调整
打开C:\devkitadv文件夹，我们可以发现devkitPPC,devkitPSP等目录，这从另一个侧面说明了gcc是可以灵活的调整编译方式，以适应各种不同的CPU
好了,我们看下devkitadv为我们带来了什么IDE……
按开始……devkitadv……是Programmers Notepad……
虽然可以进行自动的编译，但是没有整合vba等模拟器，导致运行时只能复制粘贴.gba文件然后点选vba模拟器重新打开进行测试……
也就是说，在编译上做了自动化，但是没有在运行和测试上做
没有对于资源文件清晰的显示和控制

最重要的原因……
界面太丑=v=

因此，我们需要一个整合而美观的IDE来进行开发操作。由于前一段在进行java开发感到Eclipse的环境非常非常的舒适，因此推荐统一使用Eclipse来开发此类携带机型游戏。
那么，如何让Eclipse乖乖听话来整合gcc等编译器和vba等模拟器（测试工具）呢？这就要对Eclipse
进行一些基本的配置了

二。Eclipse，VBA以及相关配置

2.1


讲了这么多，终于讲到重点
人人都是使用devkitadv做硬件开发，可是我不喜欢
也许说高手都在使用记事本，可是我觉得那只是个人风格的问题
我的个人风格TAT，我喜欢Eclipse
不仅仅是因为喜欢IBM，而且在于以后贴J2me教程的时候也要用到Eclipse

用……Eclipse来做硬件开发，特别是GBA……估计只有我一个了吧TAT
这就是所谓的苏俄风么TAT

那好，我们就来研究一下使用Eclipse怎样配制GBA开发环境吧（整合VBA模拟器哦）

嗯，Eclipse的下载地址是
http://www.eclipse.org

在下载之后，我们应该会看到一个压缩包eclipse-SDK-3.2.1-win32.zip

把它解压，会发现一个名为eclipse的文件夹，里边有个eclipse.exe
点击……运行……
会发现直接就运行了，不需要什么其他的安装过程
看来这个工具相当的绿色，eclipse这个文件夹可以随便摆

假设把eclipse这个文件夹放在D:\里边（主要依个人习惯和方便而定）
这个目录路径就是d:\eclipse
自己找个方便的位置做个快捷方式吧

再次运行eclipse，会发现里边只有J2SE的工具
因为Eclipse是以插件为主导的结构，所以没有安装针对C/C++的插件，所以还不能用eclipse来写C程序
因此，我们必须找到这个插件，这个插件的名字叫做CDT。
关于CDT的安装，老衲曾经吃过一次亏：千万不要使用网上提供的使用eclipse自动更新进行安装的方法，那样安装CDT会和现有eclipse的版本不兼容，导致eclipse不能启动。所以，我们采取手动安装的方式。

CDT的下载地址位于
http://www.eclipse.org/cdt/downloads.php

下载后解压，发现压缩包里边有两个文件夹
plugins
features
打开eclipse的目录，发现也有这两个文件夹
于是把CDT里的东西都拷进eclipse的文件夹

重新启动eclipse……
file->new->project



发现C和C++出现在列表中，说明CDT安装成功了

2.2环境变量的设置

什么是环境变量？环境变量就是path
比如在任何目录上运行ping，都可以使用，是因为在当前目录找不到的时候，会搜索环境变量里设置的目录。
我们做GBA游戏，本身编译器和库都不一样，当然不能使用CDT带的库，所以得强迫eclipse选择devkitadv里边带的一个标准库
那么在哪里设置环境变量呢？


打开控制面板->系统


选择“高级”选项卡中的最下一行“环境变量”
我们可以发现devkitadv为我们作了很多工作，现在我们要把CDT可能用到的东西补上


上下有两个窗口，一个是系统变量，一个是用户变量，这两个对于我们要做的事情没有区别
我们的任务无论在上面还是在下面，寻找lib和include


分别加上C:\devkitPro\libgba\lib


和C:\devkitPro\libgba\include

好了，环境变量设置完毕，让我们进入下一步设置：模拟器运行环境VBAadvance

2.3 vba的下载和设置

vba作为功能最强大效果最好的GBA模拟器，身为开发者的我们自然不能放过。那么，该怎么能让eclipse在该呼叫vba的时候运行vba而且可以自动载入刚编译完的程序呢？
这需要作点设置了。

先下载VBA，下载地址为
http://vba.ngemu.com/
找个自己喜欢的地方解压

打开eclipse，首先在右上角把界面布局切换为C/C++，


我们打算把VBA放在外部工具栏内，因为比较灵活，方便做设置和改进



然后，我们打开工具栏上 外部工具 的菜单，发现有一项是设置外部工具的选项，打开




首先在左上角新建处，按新建新外部工具的按钮，在program下增加一个词条
然后，依次填入
Name: VBA-dev  （当然可以自己起个好听的）
working Directory：  这里填你的工作目录
Arguments: 这里很重要，填入${project_loc}\${project_name}.gba
好了 apply保存，推出

整个IDE开发环境设置完毕。现在，我们来试试一个简单的例子吧
（一般的程序都有个hello world，但是因为字库要自己写，所以我们这里没有hello world这一说。因此用“白板”取代。等到讲字库的时候，可以让大家用任何文字来写“hello world”，只要拥有点阵字库的话。）

三。makefile 和第一个程序尝试

可能刚才已经有同学感觉有点怪怪，因为一直没有在eclipse里对编译器进行调整。其实这些工作都是由一个叫makefile的文件去做的。里边写了编译的时候，系统应该做的事情。幸运的是，我们已经拥有一个已经写好的，针对GBA和devkitadv的，万能的make文件，这个文件将在模板里一起提供给大家。


好了，现在新建一个工程文件吧

文件->新建->工程
在这里选择Standard make C


然后，输入工程名称后，直接finish


现在工程下的文件是空的


打开本帖子附带的模版包，会发现以下文件和文件夹，按右键复制


切换到eclipse，在左边资源栏按右键，粘贴到自己的project下面

好了，打开source里边的console.c，输入以下代码，嗯来个最简单的……

这种代码是不是很囧呢TAT
Copy code
int main(void) 
{

    typedef unsigned char u8;
    typedef unsigned short u16;
    typedef unsigned long u32;
    
    *(u16 *) 0x04000000 =  0x0404; 
    
    *(u16 *) 0x05000000 = 0x0000;
    *( (u16 *) 0x05000000 + 1 ) = 0xFFFF;
    
    int i = 0;
    for(i=0; i<240*160; i=i+2)
    {
        *( (u16 *) 0x06000000 + i/2 ) = 0x0001<<8 | 0x0001;
    }
    
    return 0;
    
}


解释一下我们刚才做了什么
首先，使用mode4模式（显存和屏幕一一映射的双缓冲256色图片模式，所谓双缓冲就是一张图片用显存的一半……）
打开BG2，就是第二层背景图层，注意在mode4模式下可是只有一层图层可用的讲就是BG2

往调色板内存里装填第零种颜色——黑色
往调色板内存里装填第一种颜色——白色

然后我们再240*160的屏幕中，每个点填“调色板里的第一种颜色”（就是白色）
这里还有个要讲到的东西是……虽然mode4下是8位一个点，但是机器对于显存确是16位整存整取，所以一次动就是两个点，我们把第一个点向左“移动”八位，然后和另一个点拼在一起，凑齐16位写入显存

这样效果出来就是白板一块，算是最简单的了，我们看看能不能编译


我们编译一下……

点选之后按build
