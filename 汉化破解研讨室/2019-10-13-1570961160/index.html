想成为一名破解者吗?Part IV:压缩格式

我们所接触到游戏资源文件常常经过压缩,有的时候甚至为不同类型的资源文件选择不同的算法.了解一些常用的压缩格式是成为一名成功的游戏破解者之关键所在.事实上,你需要能够仅仅通过压缩后的数据来判断这是经过怎样的算法变换,所以当你开始分析16进制转储的时候,你才会知道接下来怎么进行.
在今天的教程中,我们会介绍一些最流行的格式,如何实现,已经你该如何通过压缩后的文件进行判断.开始之前,我还是得说这个东西理论上相当困难.经过理想的压缩算法处理后的数据其实和随机噪音相差无几----不过和通常的感觉不同,这玩意可是储存了最高密度的信息.幸运的是,游戏在体积之外还有另外两个额外的要求:处理起来快,也要能被普通的码农所理解而不是需要找一个博士生[译注:而不是需要找一个死程].
这些需求意味着游戏通常使用工业标准格式,或者是来自<<小空☆教你5分钟学会压缩～ 萌之码农手册～>>之类入门书籍中相对简单快速的实现.前者通常包括一个额外的识别信息,而后者压缩的质量糟糕到看起来像是压缩文件[译注:有章可循]而不像是随机噪音.

zlib
让我们从工业标准格式zlib开始.这是一个开源压缩库,使用的是在zip和gz文件中使用到的经典算法DEFLATE.zlib工作挺快,质量也很不错.自从它作为一个完全免费的软件而编写完成后,就被运用在各种地方,其中就包括游戏压缩文件.
那么我们该如何识别它呢?0x780x9C,标准zlib库所压缩出来的文件在算法默认设置下[DEFLATE32KB缓冲区默认压缩等级]会以这两个字节开头.你也能看到0x780xDA,这意味着是使用了最大压缩等级.当你在你认为是压缩文件的开头看到这些,那就是哈哈哈哈,老娘搞定了口丫!
自从所有我所知道的现代语言都内置了zlib库后,解压缩也简单的很.在C当中,你只需要
Copy code
#include<zlib.h>
uncompress(new_buffer,&new_size, comp_buffer, comp_size);

不过在你干这事之前,先要确保你为解压缩出来的文件分配了足够多的内存,祈祷压缩文件头中包括了原始文件大小吧.这个函数返回值代表函数的执行结果,并把被解压缩的具体字节数放到new_size中.
在Python中,处理经过zlib压缩的文件也是不可思议的简单,我都不好意思写出来了
Copy code
new_data= comp_data.decode('zlib')

zlib支持任意一种内置文本编码方案,所以如果你有一个字符串的话就可以像这样展开他.如果想要自己进行一些设置的话,你也可以在你的项目中导入zlib库然后直接调用zlib的函数而不是经过高级语言的封装[译注:两个zlib库的关系就是C中的malloc和Windows中的GlobalAlloc].
压缩也很简单,我都找不到别的词来形容于是只好第三次使用简单了.C中你可以使用compress()函数或者是compress2()函数---如果你想自定义压缩级别的话.Python就是encode(’zlib’)方式.你也可以直接调用zlib库中的函数.
我不想说太多关于DEFLATE算法的内部细节,大部分时候这都不是你所需要关心的.然而有一点很重要,那就是Adler32校检.这是一个内置在zlib库中非常简单的32位校检和(类似的东西有CRC32)算法,不幸的是,有些有序也用到了这个.zlib压缩格式把这个加到压缩文件的末尾来进行错误检查.
然而,还有些游戏去掉了最后的校检和(很有可能是想要用自己的)或者把校检和移动到了压缩文件文件头中的文件索引中.这会让标准zlib解压缩函数报错,尽管结果是正确的.
所以,如果你确定这货就是zlib压缩但是总是解压报错,也许就是这个原因.你也许要自己补一个正确的校检和,或者干脆无视报错干你该干的.

游程编码(run-lengthencoding,简称RLE)
这是你愿意用的所有压缩算法当中最简单的一种.RLE常常出现在图片压缩中,有时候是一连串处理过程中的一部分.算法的核心思想就是把字节流中遇到的所有重复字节全部合并成形如重复次数+字节的样子:
Copy code
31 92 24 24 24 24 24 C5 00 00=
31 92 5*24 C5 2*00

不同的算法有不同的方式来描述这个重复次数,取决于你想让整个字节串看起来像什么.[译注:以下汉化名词非标准称呼]
标志字.你可以设定一个字节,比如0xFF作为重复字节开始的标志,然后跟着的就是一个字节表示次数和一个字节表示值,下面这个例子就是最开始那串的输出结果:
Copy code
31 92 FF 05 24 C5 FF 02 00=
31 92 5*24 C5 2*00

如果标志在你的内容中出现了,接下来的重复次数是0就好了
重复字.当你看到有字节重复1次及以上时,你就把这个字节重复1次然后跟上一个字节的后续重复次数(总次数-2)
Copy code
31 92 24 24 03 C5 00 00 00= 
31 92 24 24 3*24 C5 00 00 0*00

不过当你只重复1次时,你就要浪费一个字节来写0了
轮替型.这种方式假设你的数据是在无序的原始字节和有序的重复字节中循环,看看这个:
Copy code
02 31 92 05 24 01 C5 02 00= 
2 (31 92), 5*24, 1 (C5), 2*00

[译注:这句看不懂...只循环2字节应该是平而不是亏1字节啊...Naturally,if you have two repeated runs in a row, you’ll have to waste a byteto insert a 0-length raw sequence between them.] 一个可能的场合是你使用大量的0来间隔你需要的数据,这时候你就可以轮替出现0和原始字节.
当然你也该认识到,你所面对的算法并没有这么单纯.比如说这几个对只重复1次的字节的压缩并不高效所以他们可能干脆就用原始字节不压缩了.也有可能你的长度从3开始,因为012根本用不到嘛!也有可能使用了多字节来描述重复次数(一个马上就想得到的优化方式就是高字节首个位做标记位).
对于图片,你可能使用像素而不是字节作为循环的基本单位.这时候,就算RGB像素只重复1次也有不错的压缩效果.
那么,该怎么识别这种格式呢?一个基本原则就是所有这些变体一定会包含大量的原始字节,所以你就应该试着找找那些有规律的排列(图形格式中的RGB就很好确定),伴随着的还有一堆控制码.有一个未经压缩的原始图像版本来进行对比是很有帮助的,你可以通过截屏或者在调试器中直接从游戏内存中抽出(留到下一回再说吧[译注:坑了])

LZSS
提高压缩算法效率的一点就是关注重复数据这一整体而不是单个的字节.这种算法就会记住已经浏览过的数据,一旦发现有所重复,它就会做一个指向之前存在的重复片段的向后引用
Copy code
I love compression. This is compressed!=
I love compression. This [is ][compress]ed!=
I love compression. This [3,-3][8,-22]ed!

方括号标记出来的片段表示在之前的数据中重复出现过,所以你可以给出重复片段长度和重复片段偏移量来让解压缩程序知道该到哪里复制多少数据.包括zlib在内的很多算法都基于这种思路,但是看起来LZSS做的比它的同类们更好.
这种格式的一个特殊之处在于如何在原始数据和向后引用中切换.它在使用控制字节中的一个位来决定这一点,并且常常用1表示原始字节而0表示向后引用.这样一个控制字节就可以决定后面8个字节是啥:
[code]I love c         ompressi         on. Thi        s   [3,-3][8,-22]  ed!=
FF "I love c" FF "ompressi" FF "on. Thi"73 "s " 03 03 08 16 "ed!"
[/code]
0xFF表示”接下来是8个原始字节”,而0×73是二进制的01110011,最高位是标志位,然后从低向高读意味着2个原始字节,2个向后引用,然后是3个原始字节.[译注:第三的0FFh好像只跟随了7个字节]
在一大片控制字中想要认出这种格式,最好就是去看被压缩过的纯文本文件.如果你看FF到类似这FF样的字符FF串,文本大致可读,可又每逢第九个字节就是一个不可读取字符,那就是LZSS了.你也可以用图形文件来判断,RGB排列被控制字打断也是挺明显的.
请记住,越往文件后面看,你就越难辨认这种格式,这是由于这种算法利用文件靠前的部分作为压缩用的字典.很有可能到了最后就只剩下控制字和向后引用了.
主要的难点就在于向后引用,这点上有许多变种.最流行的一种以4KB为单位进行分段,然后16位的向后引用包括了12位的偏移量和4位的长度.然而,这个长度是真实长度还是减了3[译注:对长度进行处理,使第三方不易理解]?是相对地址还是绝对地址?控制字是从高位还是从低位开始读取?我曾看过这样一个例子:在一个小段中使用1字节的偏移量;然后这个偏移量指向一个2字节,位于一个中段的偏移量;最后这个2字节偏移量又指向一个3字节,位于一个大段的偏移量;再来才是想要的数据.所以你需要自己做点实验,看看特定的游戏到底是怎么处理的.
很奇妙的一点就是你可以(如果允许的话)让那些压缩数据重叠起来,实现的方法就是让长度大于偏移量的绝对值:
[code]This is freeeeeeeaky!
=This [is ]fre[eeeeee]aky!
=This [3,-3]fre[6,-1]aky!
[/code]
向后引用[6,-1]能正常工作的原因是解压缩一次只拷贝1个字节:首先指针指向了最原始的e,复制.此时指针指向复制出来的第一个e,继续向下复制.如此反复6次即可.请在你实现你自己的算法时留心这一点,因为这可以使你无法处理某些特定种类的数据,而且也不是所有的游戏都能理解这种引用方式.所以不要只为你自己压缩.

霍夫曼编码(Huffmanencoding)
Fromone point of view, this is easier than other algorithms since it onlyworks on single bytes (or symbols, in general) at a time, but it’salso more tricky since the compressed data is a bitstream rather thanbeing easy-to-digest bytes and control codes.
It works by figuringout the frequencies of all the bytes in a file, and encoding the morecommon ones with fewer than 8 bits, and the less common ones withmore than 8, so you end up with a smaller file on average. This isvery closely related to concepts of entropy, since each symbolgenerally gets encoded with a number of bits equal to its own entropy(as determined by its frequency).
Let’s be specific. Considerthe string “abracadabra”. The letter breakdown is:
a : 5/11 ~1.14 bitsb : 2/11 ~ 2.46 bitsc : 1/11 ~ 3.46 bitsd : 1/11 ~ 3.46bitsr : 2/11 ~ 2.46 bits[/pre]Where I’ve given the number of bitsof entropy each frequency corresponds to (i.e. if you have a 25%chance of having a certain letter, it has a 2-bit entropy since youneed to give one of 4 values, say 00,to specify it out of the other 75% of possibilities: 01, 10, 11).Unfortunately we can’t use fractional bits, so we may have to roundup or down from these theoretical values.
How do we choose theright codes? Well, the best way is to build up a tree, starting fromthe least-likely values. That is, we treat, say, “c or d” as asingle symbol with a frequency of 2/11, and say that if we get thatfar we know we can just spend one extra bit to figure out whether wemean c or d:
a : 50=c + 1=d : 2b : 2r : 2[/pre]Then we continuedoing the same thing. At each step we combine the two least-weightitems together, adding one bit to the front of their codes as we go.In the case of ties, we pick the ones with shorter already-assignedcodes, or alphabetically first values:
a : 50=b + 1=r : 40=c + 1=d: 200=b + 01=r + 10=c + 11=d : 6a : 5000=b + 001=r + 010=c + 011=d +1=a : 11[/pre]So the codes we end up with are:
a : 1b : 000c :010d : 011r : 001[/pre]You will notice an excellent property of thesecodes: they are not ambiguous. That is, you don’t have 1 for‘a’ and 100 for‘b’… as soon as you hit that first 1,you know you can stop and go on to the next symbol without needing toread any more. Therefore, “abracadabra” just gets encoded as:
ab r a c a d a b r a1 000 001 1 010 1 011 1 000 001 1= 1000001101010111 00000110= 83 57 06[/pre]We’ve compressed 88 bits (11bytes) down to 23 bits (just under 3 bytes). Almost always the bitsare packed most- to least-significant in a byte.
One subtlety isthe exact method of tree creation, which assigns the codes. Themethod described above is “canonical”, but sometimes games willuse their own idiosynchratic methods which you will have to matchexactly to avoid getting garbage.
How do you recognize this in adata file? Well, the decompressor needs to know the codes, and theeasiest way to specify this is to give it the frequencies (or moreeasily, the bit weights) of the values so it can construct its owntree.
Therefore, the compressed data will usually start with, say,a 256-element table of bit weights. So if you see 256 bytes of 0506 08 07 0C 0B 06 —values that are around 8 plus or minus a few — followed byhorrendous random junk, you’re probably looking at Huffmanencoding.
Sometimes instead of bit weights you’ll have theactual frequency counts instead, which might need to have amulti-byte encoding scheme if they’re above 256. In that case,you’re mainly looking for a few hundred bytes of “stuff”followed by a sharp transition to much more random data.
Otheralgorithms
Needlessto say, the algorithms covered here are not the full range ofcompression formats out there. I’ll just briefly mention someothers in case you run across them, though I haven’t really seenthem in the wild.
Arithmeticencoding. Thisis vaguely related to Huffman encoding, in that you are workingstrictly with single bytes (or symbols) and trying to stuff the mostfrequent ones into fewer bits. However, instead of being restrictedto an integral number of bits for each one, here you are allowed tobe fractional on average.
This works by breaking up the numericalinterval [0,1) into subranges corresponding to each symbol: the morecommon symbols correspond to larger ranges, in proportion to theirfrequency. You start with [0,1), and the first byte resticts you tothe subrange for that symbol. Then the second byte restricts you to asub-subrange, the third byte a sub-sub-range, etc. Your final encodeddata is any single numerical value inside the tiny range you end upin: just pick the number in that range you can represent in the leastnumber of bits as a binary fractional value.
Needless to say thereare some good tricks for implementing this without usingludicrously-high-precision math, but I won’t go into that.
LZ77(Lempel-Ziv ’77)
Thisis the core of the zlib deflate algorithm, but you’ll sometimes seevariants outside of that standard, so it’s useful to know a littleabout. It’s basically a combination of standard back-references asin LZSS, plus Huffman encoding. You just treat the back-referencecommand “copy 8 bytes” as a special symbol, like a byte value of256+8=264.
Then, with this mix of raw data bytes andback-reference symbols, you run it through a Huffman encoding to getthe final compressed output. Typically you will do somethingdifferent with the back-reference offsets: either leave them as rawdata, or encode them in their own separate Huffman table.
LZW(Lempel-Ziv-Welch)
Whentaught correctly, this is an algorithm with a mind-blowing twist atthe end. As it runs through the file, it builds up an incrementaldictionary of previously-seen strings and outputs codes correspondingto the dictionary entries. And then, at the end, when you start towonder how to encode this big dictionary so the decompressor can useit to make sense of the codes, you just throw the dictionary away.Cute.
Of course it turns out that things are cleverly designed sothat the decompressor can build up an identical dictionary as it goesalong, so there’s no problem. This algorithm was patent-encumberedfor a while, so it didn’t get as widely adopted as it mightotherwise have been, but you might start seeing more of it thesedays.
Conclusion
I’vefocused here on lossless general-purpose compression: the sorts ofthings that are done to data at the archive level. There is also astage below this, where data can be compressed before even being putinto the archive: making raw images into JPEGs, PNGs, or othercompressed image formats, and converting sounds to MP3s, OGGs, and soforth. In many cases those compression steps are just a lossyapproximation to the original data, which is okay for graphics andsounds but bad for other files.
In a later installment, I’ll betackling image formats in particular in more detail, since you willtend to run across custom ones a lot, some of which includeimage-specific processing steps (like, say, subtracting pixels fromtheir neighbors) which wouldn’t make a lot of sense in a moregeneral-purpose compression algorithm. Encryption is another latertopic, since sometimes that will keep you from being able torecognize compressed data for what it is.
And naturally, if you’verun across other general compression algorithms used in games you’velooked at, please mention them in the comments, since I don’t pretend to have investigated all the games out there… I’m stillbeing surprised all the time.
